# 질문 해결 노트

교재 단위로 공부하지 않을 때 생기는 짧은 질문도 그냥 흘리지 않고 남긴다. 이 문서는 "그때 왜 막혔는지", "어떻게 이해가 풀렸는지", "나중에 블로그나 면접 답변으로 어떻게 꺼낼지"를 짧게 정리하는 필드 노트 아카이브다.

# PART 1. 브라우저와 캐시

## 1. DevTools가 열려 있을 때만 네트워크를 기록하는 이유

### 상황

- 새 창이 뜨는 흐름을 확인하려고 했는데, DevTools를 늦게 열면 그 직전 요청은 네트워크 탭에 남아 있지 않았다.
- 겉으로 보면 "브라우저가 요청을 이미 보냈는데 왜 나중에 못 보여주지?"라는 의문이 생긴다.

### 핵심 이유

DevTools는 브라우저 내부 로그를 아무 때나 들춰보는 창이 아니라, 탭에 attach한 뒤 이벤트를 받아보는 디버깅 클라이언트다. DevTools가 열리는 순간 Chrome DevTools Protocol(CDP)이 해당 탭에 붙고, 그 시점부터 네트워크 이벤트 스트림을 구독한다.

브라우저는 성능 비용 때문에 모든 네트워크 요청을 평소부터 영구 저장하지 않는다. 즉, 구독자가 없는 동안 발생한 요청은 "나중에 볼 수 있게 임시 저장"되는 것이 아니라, 애초에 DevTools용 기록으로는 남지 않는다.

### 기억할 포인트

1. 네트워크 요청 발생과 DevTools 기록 시작은 같은 사건이 아니다.
2. DevTools는 과거 이력을 복원하는 뷰어라기보다, 붙는 시점부터 이벤트를 받는 관찰자에 가깝다.
3. 새 창 초기 요청까지 꼭 봐야 하면 DevTools를 미리 열어 두거나, Preserve log/자동 열기 흐름을 준비해야 한다.

### 블로그나 면접으로 꺼낼 문장

- "관찰 도구가 항상 녹화 중이라고 가정하면 디버깅이 꼬인다. 실제로는 attach 이후부터 이벤트를 구독하는 구조인 경우가 많다."
- "DevTools를 늦게 연 뒤 과거 요청이 안 보이는 건 버그라기보다 도구 연결 시점의 한계다."

## 2. 포털을 새로고침해야 새 배포가 보였던 이유

### 상황

- Project A는 배포와 무효화가 끝났고, A 자체 URL도 맞았다.
- 그런데 여러 프로젝트 URL을 모아 두는 포털에서 A로 들어가면 예전 결과물이 보였고, 포털 자체를 강력 새로고침한 뒤에야 새 배포가 정상적으로 보였다.

### 핵심 이유

핵심은 "URL이 같아도 그 URL을 여는 주체가 다르면 결과가 달라질 수 있다"는 점이다.

포털이 단순한 정적 링크가 아니라, 자체 JavaScript 런타임으로 라우팅하거나 외부 앱 청크 경로를 동적으로 조립하는 구조라면 포털의 캐시된 JS가 오래된 참조를 쥐고 있을 수 있다. 예를 들어 포털 번들 안에 Project A의 예전 청크 해시나 진입 경로가 남아 있으면, A의 `index.html`만 무효화해도 포털은 계속 구버전 자산을 향해 갈 수 있다.

그래서 문제는 A URL 하나가 아니라, "A로 들어가는 입구 앱"의 캐시된 런타임이었다. 포털을 강력 새로고침하면 포털의 새 JS가 내려오고, 그제서야 새 해시 경로나 새 라우팅 규칙으로 A를 다시 연다.

### 어디를 의심해야 하나

1. 포털이 SPA 라우팅, module federation, iframe bootstrap, 동적 script 주입 중 무엇으로 외부 앱을 여는가
2. 포털의 `index.html`과 번들 JS에 `Cache-Control`이 어떻게 설정되어 있는가
3. 포털 또는 Project A 쪽 Service Worker가 중간에서 예전 응답을 가로채고 있지 않은가
4. 무효화 대상이 A의 HTML만이었는지, 포털의 런타임/manifest까지 포함했는지

### 블로그나 면접으로 꺼낼 문장

- "같은 URL이라도 진입 주체의 캐시된 런타임이 다르면 사용자가 보는 결과는 달라질 수 있다."
- "배포 문제를 볼 때는 대상 앱만 보지 말고, 그 앱으로 들어가는 포털이나 셸의 캐시 전략까지 같이 봐야 한다."

## 3. React 프로젝트에서 Node 버전을 관리하는 방법

### 특정 버전으로 시작할 때 순서

가장 안전한 전제는 "프로젝트를 만들기 전에 Node 버전을 먼저 고정한다"는 것이다. React 자체보다 먼저 Node 런타임이 정해져 있어야, 생성 도구와 패키지 설치 결과가 처음부터 일관되게 나온다.

1. Node 버전 매니저를 먼저 쓴다. `nvm`, `fnm`, `asdf`, `Volta` 중 하나면 충분하다.
2. 원하는 버전을 설치한다. 예: `nvm install 22.14.0`
3. 그 버전을 활성화한다. 예: `nvm use 22.14.0`
4. 프로젝트 폴더에 `.nvmrc`를 저장한다. 예: `22.14.0`
5. 그 상태에서 프로젝트를 생성한다. 예: `pnpm create vite`, `pnpm create next-app`
6. `package.json`에 `engines.node`를 적어 둔다. exact version이나 major range를 팀 규칙으로 정하면 된다.
7. `node -v`로 실제 버전이 맞는지 확인하고, 그다음에 의존성을 설치한다.
8. CI, Docker, 배포 환경도 같은 Node 버전으로 맞춘다. 로컬만 맞아도 팀 전체는 안 맞을 수 있다.

예를 들어 협업 프로젝트에서는 보통 아래 두 파일을 같이 남긴다.

```text
.nvmrc
22.14.0
```

```json
{
  "engines": {
    "node": "22.14.0"
  }
}
```

핵심은 "내가 가진 특정 버전으로 시작하고 싶다"면, 프로젝트 생성 전에 그 버전으로 셸 자체를 먼저 바꾸고 시작해야 한다는 점이다.

### 그냥 최신 버전으로 시작할 때

여기서 "최신"은 두 가지로 나뉜다.

1. 가장 최신 current 버전
2. 가장 최신 LTS 버전

실무에서는 보통 current보다 LTS가 안전하다. 생태계 호환성과 CI 안정성이 더 좋기 때문이다.

- 실험용으로 정말 최신 current를 쓰고 싶으면: `nvm install node`
- 협업용으로 최신 안정 계열을 쓰고 싶으면: `nvm install --lts`

중요한 건 최신 버전으로 시작하더라도, 한 번 정했으면 바로 `.nvmrc`와 `engines`에 적어 프로젝트 버전으로 고정해야 한다는 점이다. 그렇지 않으면 "오늘의 최신"과 "다음 달의 최신"이 달라져서 재현성이 무너진다.

### 이미 만든 프로젝트의 Node 버전을 바꾸고 싶을 때

1. 먼저 바꾸려는 버전을 현재 프레임워크와 주요 패키지가 지원하는지 확인한다.
2. 새 버전을 설치하고 활성화한다. 예: `nvm install 24.1.0`, `nvm use 24.1.0`
3. `.nvmrc`와 `package.json`의 `engines.node`를 함께 수정한다.
4. CI 설정, Dockerfile, 배포 런타임도 같은 버전으로 올린다.
5. `node_modules`를 기준 버전에 맞게 다시 설치한다. major 변경이면 재설치가 특히 안전하다.
6. `pnpm lint`, `pnpm test`, `pnpm build`까지 다시 돌려 런타임 차이로 깨지는 곳이 없는지 확인한다.

버전 변경에서 자주 놓치는 부분은 "로컬 Node만 바꾸고 끝내는 것"이다. 실제로는 설치 환경, CI, 배포 환경, 네이티브 모듈 재빌드까지 같이 움직여야 제대로 바뀐 것이다.

### 왜 팀원들 Node 버전이 다르면 문제가 생기나

1. 어떤 사람은 설치가 되고 어떤 사람은 `engines` 오류나 문법 오류로 시작부터 막힌다.
2. `esbuild`, `sharp`, `@swc/core`, `better-sqlite3` 같은 네이티브/바이너리 의존성이 버전별로 다르게 동작할 수 있다.
3. 로컬에서는 되는데 CI나 다른 팀원 환경에서는 테스트, 빌드, lint 결과가 달라질 수 있다.
4. Node 내장 API 차이 때문에 `fetch`, `Web Crypto`, `ESM/CJS`, `fs` 동작에서 미묘한 차이가 난다.
5. 결국 "내 컴퓨터에서는 되는데?" 상황이 늘어나서 디버깅 비용이 커진다.

그래서 협업 프로젝트에서는 최소한 이 네 가지를 맞추는 편이 좋다.

1. `.nvmrc` 또는 `volta` 같은 버전 고정 파일
2. `package.json`의 `engines.node`
3. CI의 Node 버전
4. 배포 환경의 Node 버전

### 기억할 포인트

1. Node 버전은 프로젝트 생성 전에 먼저 고정하는 게 가장 깔끔하다.
2. "최신"으로 시작하더라도 결국 프로젝트 버전으로 즉시 pin 해야 한다.
3. 버전 변경은 로컬 셸만이 아니라 설치, CI, 배포까지 한 세트로 봐야 한다.
4. 팀원이 서로 다른 버전을 쓰면 결국 재현성과 디버깅 비용에서 손해를 본다.

### 블로그나 면접으로 꺼낼 문장

- "React 프로젝트의 첫 의존성은 React가 아니라 Node 런타임이다. 생성 전에 Node 버전부터 고정해야 설치 결과가 재현 가능해진다."
- "협업에서는 최신 버전을 쓰는 것보다, 모두가 같은 버전을 쓰는 것이 더 중요하다."
