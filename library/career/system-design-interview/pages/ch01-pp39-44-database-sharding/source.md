# 1장 · 39~44p 데이터베이스 샤딩과 1장 마무리

## 수직적 확장

> p38에서 시작한 `수직적 확장` 문장이 p39로 이어진다.

수직적 규모 확장은 기존 데이터베이스 서버에 더 많은 고성능 자원(CPU, RAM, 디스크 등)을 증설하는 접근이다.

보이는 주요 단점:

- 데이터베이스 서버 하드웨어에는 한계가 있다.
- SPOF로 인한 위험성이 크다.
- 고성능 서버로 갈수록 비용이 많이 든다.

## 수평적 확장

데이터베이스의 수평적 확장은 샤딩이라고도 부르며, 더 많은 서버를 추가해 성능을 향상시키는 접근이다.

### 그림 1-20 요약

- 수직적 규모 확장: 한 서버에 CPU, RAM, 디스크 같은 자원을 더 올린다.
- 수평적 규모 확장: 서버 수를 늘려 데이터를 나눈다.

샤딩은 대규모 데이터베이스를 샤드라고 부르는 작은 단위로 분할하는 기술이다. 모든 샤드는 같은 스키마를 쓰지만, 샤드에 보관되는 데이터 사이에는 중복이 없다.

### 그림 1-21 요약

- `user_id % 4` 결과로 0, 1, 2, 3번 샤드 중 어디에 저장할지 정한다.

### 그림 1-22 요약

- 샤드 0: `user_id` 0, 4, 8, 12...
- 샤드 1: `user_id` 1, 5, 9, 13...
- 샤드 2: `user_id` 2, 6, 10, 14...
- 샤드 3: `user_id` 3, 7, 11, 15...

## 샤딩 키와 운영 문제

샤딩 전략을 구현할 때 고려해야 할 가장 중요한 것은 샤딩 키를 어떻게 정하느냐이다.

발췌:

> 샤딩 키는 파티션 키(partition key)라고도 부르는데, 데이터가 어떻게 분산될지 정하는 하나 이상의 칼럼으로 구성된다.

운영 중 고려할 문제:

- 데이터의 재 샤딩
- 유명인사 문제 또는 hotspot key 문제
- 조인과 비정규화

발췌:

> 이를 해결하는 한 가지 방법은 데이터베이스를 비정규화하여 하나의 테이블에서 질의가 수행될 수 있도록 하는 것이다.

### 그림 1-23 요약

- 사용자 단말은 DNS, CDN, 로드밸런서를 거쳐 웹 서버로 간다.
- 웹 서버는 샤딩된 데이터베이스, 캐시, 메시지 큐, 작업 서버, NoSQL과 연결된다.
- 로그, 메트릭, 모니터링, 자동화 도구가 시스템 관측과 운영을 돕는다.

## 백만 사용자, 그리고 그 이상

1장은 시스템 규모 확장이 지속적이고 반복적인 과정임을 정리한다. 핵심 요약은 무상태 웹 계층, 다중화, 캐시, 다중 데이터 센터, CDN, 데이터 계층 샤딩, 계층별 독립 서비스 분리, 모니터링과 자동화다.
