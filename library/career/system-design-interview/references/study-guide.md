# 대규모 시스템 설계 기초 스터디

이 트랙은 책 내용을 그냥 요약해서 끝내는 곳이 아니다. 앞으로 네가 계속 보내 줄 캡처와 해석을 바탕으로, OCR 텍스트 아카이브와 학습 코멘트, 오개념 교정, 용어 풀이, 예시, 근거 링크를 한 자리에 계속 누적하는 작업대다.

프론트엔드 개발자인 네가 시스템 설계를 공부하는 이유는 "백엔드 지식을 억지로 외우기 위해서"가 아니다. 화면에서 보이는 버그와 느림, 캐시 꼬임, 검색 품질, 실시간 업데이트, 알림 지연, 업로드 실패, 피드 정렬 문제의 뒤에는 거의 항상 시스템 설계 판단이 깔려 있다. 즉, 이 책은 프론트엔드를 버리고 백엔드로 가기 위한 책이 아니라, 프론트엔드 관찰력을 더 넓은 시스템 감각으로 연결하기 위한 책으로 읽는다.

원문 아카이브 원칙상, 2026-04-22에 전달된 목차 캡처는 스크린샷 자체를 저장하지 않고 OCR 텍스트만 별도 보조 파일인 `library/career/system-design-interview/references/table-of-contents-archive.md`에 남긴다. 이 스터디 문서에서는 그 텍스트를 바로 읽을 수 있게 다시 실어 두고, 이어서 왜 이 순서로 공부하는지와 프론트엔드 관점의 우선순위를 덧붙인다.

# PART 1. 이 트랙을 어떻게 쓸지

## 1. 이 트랙의 목적

이 트랙의 목적은 세 가지다.

1. 책 문장을 빠뜨리지 않고 아카이브한다.
2. 책이 생략한 맥락, 잘못 이해하기 쉬운 부분, 실무에서 더 중요한 연결 지점을 친절하게 보강한다.
3. 네가 보낸 생각과 해석을 나중에 다시 찾기 쉽게 질문 기록으로 남긴다.

여기서 가장 중요한 원칙은 "프론트엔드 중심"과 "풀스택 확장"을 동시에 잡는 것이다. 우선순위는 분명히 프론트엔드다. 하지만 CDN, 캐시, 메시지 큐, 분산 ID, 검색, 알림, 피드, 채팅, 스토리지 같은 주제는 결국 프론트엔드 경험 품질과 바로 연결되므로, 백엔드라고 해서 밀어내지 않는다.

## 2. 앞으로 캡처를 보내면 이렇게 반영한다

앞으로 장별 캡처를 보내면 아래 순서로 계속 누적한다.

1. 스크린샷 이미지는 보관하지 않고 OCR 텍스트만 추출한다.
2. 같은 캡처를 여러 번 대조해 오탈자, 문단 경계, 페이지 번호를 교차검증한다.
3. 보이는 텍스트를 빠뜨리지 않고 원문 아카이브로 정리한다.
4. 문맥이 비거나 책 설명이 짧은 부분에는 보강 설명을 붙인다.
5. 네가 함께 보낸 해석과 생각은 질문 기록으로 따로 남긴다.
6. 맞는 부분은 무엇이 좋았는지 짚고, 틀린 부분은 왜 틀렸는지 정확히 교정한다.
7. 이해가 어려운 용어는 쉬운 한국어와 원어를 함께 적는다.
8. 추상적인 설명은 프론트엔드에서 바로 감이 오는 예시로 바꿔 준다.
9. 관련 공식 문서, 논문, 빅테크 자료, GitHub, 커뮤니티 링크를 같이 누적한다.
10. 질문 원문은 줄바꿈과 목록을 보존해 `질문 원문` 블록에 남긴다.
11. 질문 자체의 문장, 전제, 범위를 더 좋게 만들 수 있으면 `질문 다듬기` 블록을 반드시 붙인다.

### OCR 반영 원칙

- 이미지는 저장하지 않는다.
- OCR 텍스트만 남긴다.
- 확신이 낮은 줄은 별도 메모로 표시하고 다시 대조한다.
- 최종 화면은 책 레이아웃 복제가 아니라 읽기 좋은 텍스트 레이아웃을 우선한다.

질문 기록에는 단순히 "정답"만 남기지 않는다. 무엇이 막혔는지, 왜 그렇게 해석했는지, 어느 문장/그림/예시에서 틀어졌는지를 함께 남겨야 나중에 다시 봐도 복습 가치가 생긴다.

### 답안 토글 사용법

각 페이지의 `읽으면서 생각하기`는 기본적으로 닫힌 토글이다. 먼저 질문만 보고 스스로 답을 만든 뒤 열어 본다. 토글 안에는 `모범 답안`과, 프론트엔드 면접에서 어필할 수 있는 경우에는 `프론트 면접 어필`이 함께 나온다.

모범 답안은 암기용 문장이 아니다. 좋은 답변의 골격을 보여 주는 worked example이다. 답을 열어 본 뒤에는 1) 내 답에서 빠진 구성 요소, 2) 내가 말하지 못한 trade-off, 3) 프론트엔드 경험 품질과 연결할 수 있는 지점을 표시한다.

## 3. 프론트엔드 개발자가 특히 챙길 관점

- 캐시: 브라우저 캐시, CDN 캐시, API 캐시, 서버 캐시가 각각 어디서 적용되는지 구분해야 배포 장애와 stale UI를 읽을 수 있다.
- 무상태 웹 계층: 세션을 서버 메모리에 붙여 두는 설계가 왜 확장성에 약한지 이해해야 로그인, 장바구니, sticky session 문제를 설명할 수 있다.
- 메시지 큐와 이벤트: 프론트엔드가 직접 큐를 다루지 않더라도, 알림/피드/채팅/업로드 완료 같은 비동기 UX는 대부분 큐나 이벤트 소비 흐름 위에서 만들어진다.
- 검색과 추천: 자동완성, 피드, 랭킹은 "API 하나 호출하면 나온다"가 아니라 인덱싱, fanout, 랭킹, 캐시 전략이 합쳐진 결과라는 감각이 필요하다.
- 실시간성: 채팅, 알림, 진행률, 업로드 상태, 협업 편집은 전부 "언제 push 하고 언제 pull 하는가"의 문제다.
- 운영 지표: 로그, 메트릭, 알림, 재시도, 백오프는 백엔드 전용이 아니라, 프론트 장애를 빠르게 설명하고 재현하고 완화하기 위한 기반이다.

## 4. 내가 앞으로 코멘트할 때 지킬 원칙

- 애매한 칭찬보다, 무엇을 정확히 이해했고 무엇을 놓쳤는지 분리해서 말한다.
- "틀렸다"에서 끝내지 않고, 왜 그렇게 오해하기 쉬운지도 설명한다.
- 질문 자체에 모순, 과한 전제, 범위 혼동, 문장 흐름 문제가 있으면 `질문 다듬기`로 바로잡는다.
- 용어를 그대로 반복하지 않고 쉬운 한국어로 다시 풀어쓴다.
- 프론트엔드에서 바로 체감되는 예시를 먼저 들고, 그 뒤에 서버/인프라 관점으로 확장한다.
- 공식 문서나 논문이 더 강한 근거가 될 수 있으면 커뮤니티 글보다 먼저 참고한다.

## 5. 지금부터는 이렇게 저장한다

- 한 번에 책 한 장 전체를 기다리지 않는다. 네가 `여기까지 읽고 보내보자`라고 보낸 분량을 하나의 읽기 배치로 저장한다.
- 첫 실제 적재 배치는 `1장 · 14~15p 도입과 단일 서버`다.
- 두 번째 적재 배치는 `1장 · 16p 요청의 출발점과 클라이언트 종류`다.
- 세 번째 적재 배치는 `1장 · 17p 데이터베이스 분리와 선택`이다.
- 네 번째 적재 배치는 `1장 · 18p NoSQL 선택과 규모 확장 방식`이다.
- 다섯 번째 적재 배치는 `1장 · 19~21p 로드밸런서와 데이터베이스 다중화`다.
- 여섯 번째 적재 배치는 `1장 · 24~26p 캐시와 CDN 도입`이다.
- 일곱 번째 적재 배치는 `1장 · 27~29p CDN 동작과 무효화`다.
- 여덟 번째 적재 배치는 `1장 · 30~32p 무상태 웹 계층`이다.
- 아홉 번째 적재 배치는 `1장 · 33~35p 데이터 센터`다.
- 열 번째 적재 배치는 `1장 · 35p 메시지 큐 도입`이다.
- 열한 번째 적재 배치는 `1장 · 36p 메시지 큐 확장 예시`다.
- 열두 번째 적재 배치는 `1장 · 36~38p 로그, 메트릭 그리고 자동화`다.
- 열세 번째 적재 배치는 `1장 · 38p 데이터베이스 규모 확장 도입`이다.
- 현재까지 이 열세 배치에는 OCR 원문, 그림 레이블, 쉬운 해설, 용어, 네가 실제로 막힌 질문 29개가 함께 연결돼 있다.
- 앞으로는 스크린샷 한 장이 완전히 준비되지 않았더라도, 네가 책 문장을 `""`로 직접 발췌하고 그 아래에 의문점·사고흐름·질문을 적어 보내면 우선 학습 기록과 복습 큐에 반영한다. 이후 같은 위치의 캡처가 더 모이면 해당 질문을 정식 배치 페이지로 다시 연결한다.
- 앞으로도 같은 방식으로 `트랙 개요 -> 읽을 문서 -> 배치 페이지 -> 학습 기록` 흐름이 계속 누적된다.

# PART 3. 장별 학습 포인트

## 1장 사용자 수에 따른 규모 확장성

- 프론트엔드 초점: CDN, 캐시, 무상태 웹 계층, 데이터 센터, 로그/메트릭이 사용자 체감 속도와 배포 안정성에 어떻게 연결되는지 본다.
- 풀스택 확장: 데이터베이스 분리, 메시지 큐, 자동화, 다중 데이터센터 운영을 왜 함께 봐야 하는지 잡는다.
- 짧은 예시: 새 배포가 되었는데 일부 사용자만 구버전을 본다면, 프론트 화면 문제처럼 보여도 실제 원인은 CDN 캐시나 셸 앱 캐시일 수 있다.

### 첫 배치에서 바로잡은 오해 1. 배포 단위와 실행 배치는 다르다

- `단일 서버`는 CI/CD가 하나라는 뜻이 아니다.
- 책이 말하는 것은 `어디에 배포 버튼을 몇 번 누르느냐`가 아니라 `실제로 어느 머신에서 어떤 프로세스들이 함께 도느냐`다.
- 프론트 빌드 결과물, 앱 서버, MySQL, Redis를 각각 따로 배포하더라도 결국 같은 VM 한 대에서 함께 실행되면 책의 문맥에서는 여전히 `한 서버 안에 다 같이 올라간 구조`, 즉 단일 서버다.
- 반대로 프론트는 CDN, 백엔드는 앱 서버 팜, DB는 별도 관리형 인스턴스로 분리되어 있으면 배포 파이프라인이 단순해도 이미 단일 서버가 아니다.

짧은 예시:
한 대의 EC2 안에 `nginx`, `node app`, `postgres`, `redis`가 같이 돈다.
정적 파일은 따로 업로드하고, 백엔드는 따로 재배포하고, DB는 migration도 따로 돌릴 수 있다.
그래도 실제 실행 위치가 한 머신이므로 single server(한 서버 안에 다 같이 올라간 구조)다.

조금 더 정확히 나누면:

- `배포 단위`: 프론트 정적 파일 업로드, 백엔드 API 릴리스, DB migration 실행처럼 따로 관리되는 릴리스 절차
- `실행 배치`: nginx, 앱 프로세스, DB, Redis가 실제로 어느 머신에서 돌아가는가
- 책의 `단일 서버` 판정 기준은 첫째가 아니라 둘째다.

AWS 식으로 비교하면:

- `단일 서버` 예시
  - 하나의 EC2 안에 `nginx + node app + postgres + redis`
  - 프론트 결과물은 그 EC2의 디스크에 올라가고, API도 같은 EC2에서 응답하고, DB와 캐시도 같은 EC2 안에서 돈다.
- `이미 단일 서버가 아닌` 예시
  - 프론트: `S3 + CloudFront`
  - 앱 서버: `EC2`
  - 데이터베이스: `RDS`
  - 캐시: `ElastiCache`
  - 이 경우는 실행 위치 자체가 이미 여러 인프라로 갈라져 있으므로 단일 서버가 아니다.

캐시는 어디 있느냐는 질문에는 이렇게 답하면 된다:

- 브라우저 캐시: 사용자 기기 안
- CDN 캐시: CloudFront 같은 사용자 가까운 CDN 서버
- 리버스 프록시 캐시: nginx 같은 웹 서버 앞단 캐시
- 애플리케이션 메모리 캐시: 앱 프로세스 내부 메모리
- 외부 캐시: Redis 또는 ElastiCache

즉 `캐시도 같이 배포한다`는 말은 책 문맥에서는 `같은 머신에 Redis 같은 캐시 프로세스까지 같이 올려 둔 단순 모델`을 가리키는 것이지, 캐시가 언제나 프론트와 같은 방식으로 배포된다는 뜻이 아니다.

### 첫 배치에서 바로잡은 오해 2. 프론트도 원래 하던 최적화인데 왜 대규모에서 특별해지나

- 네 감각이 완전히 틀린 건 아니다. 요청 빈도 조절, 캐시, 코드 스플리팅은 원래 모든 프로젝트에서 하는 기본기다.
- 하지만 대규모에서는 이 기본기가 `있으면 좋은 최적화`가 아니라 `용량·비용·장애 전파`를 좌우하는 설계 문제가 된다.
- 차이는 기술 종류보다 `파급 효과의 크기`다.

여기서 말하는 `시스템 부하의 모양`은 단순히 총 요청 수만 뜻하지 않는다.

- 요청률: 초당 몇 번 호출되는가
- 버스트: 짧은 시간에 얼마나 몰리는가
- 중복: 같은 데이터를 여러 번 다시 가져오는가
- fan-out: 프론트 한 번의 동작이 백엔드 여러 서비스 호출로 번지는가. 즉 요청 하나가 뒤에서 여러 요청으로 퍼지는가
- payload 크기: 한 번의 요청과 응답이 얼마나 무거운가
- cache hit ratio: CDN이나 브라우저 캐시가 실제 요청을 대신 처리해 준 비율이 얼마나 되는가

즉 프론트는 `서버가 준 결과를 화면에 그리는 층`에서 끝나지 않는다. 사용자가 어떤 타이밍으로, 어떤 패턴으로, 얼마나 무겁게 시스템을 두드리게 만들지까지 결정한다.

짧은 예시:
검색창에서 키 입력마다 요청을 날리는 UI는 작은 서비스에서는 좀 비효율적인 수준일 수 있다.
하지만 대규모에서는 같은 실수가 검색 API QPS를 폭증시키고, 캐시 미스와 재시도까지 겹치면 장애 증폭점이 된다.

또 다른 예시:
번들 크기 200KB 증가는 작은 프로젝트에서는 로딩이 조금 느린 문제일 수 있다.
하지만 모바일 사용자가 많고 지역이 여러 곳이면 이 선택은 전송 비용, 초기 렌더링 지연, 이탈률 증가로 이어진다.

대규모에서 더 잘 와닿는 예시는 이렇다:

- 20만 명이 동시에 쓰는 검색 화면에서 사용자가 평균 8글자를 입력한다.
- debounce 없이 키 입력마다 자동완성 API를 호출하면 짧은 시간에 최대 160만 번의 추가 요청이 생긴다.
- 검색 API 뒤에 추천 서비스, 철자 교정, 로그 적재가 붙어 있으면 프론트 한 번의 입력이 백엔드 여러 곳으로 fan-out(요청 하나가 여러 요청으로 퍼지는 현상) 된다.
- 여기서 timeout(응답 제한 시간) 후 즉시 재시도까지 붙으면, 서버가 느려지는 순간 프론트가 오히려 버스트를 더 키우는 self-DDoS(자기 시스템을 스스로 몰아치는 과부하)가 된다.

`사용자 체감 품질`도 더 구체적으로 보면:

- 같은 300ms API라도 화면이 즉시 반응하면 사용자는 빠르다고 느낀다.
- 반대로 API는 빨라도 JS 파싱과 hydration(서버가 그린 HTML에 브라우저 JS가 다시 기능을 붙이는 과정)이 무겁고 main thread(브라우저가 화면과 JS를 주로 처리하는 실행 흐름)가 막히면 입력이 씹히고 스크롤이 끊긴다.
- CDN 캐시가 잘 맞으면 첫 페인트가 빨라지고, 캐시가 어긋나면 같은 화면도 어떤 사용자는 즉시 뜨고 어떤 사용자는 오래 기다린다.
- 즉 사용자가 느끼는 품질은 서버 응답 시간만이 아니라 브라우저 실행 비용, 캐시 적중률, 렌더링 전략까지 합쳐져서 만들어진다.

핵심 문장:
프론트엔드가 대규모에서 특별한 이유는 새로운 종류의 일을 해서가 아니라, 원래 하던 선택 하나하나가 훨씬 큰 부하와 비용과 장애 반경을 만들기 때문이다.

한 번 더 냉정하게 정리하면:

- 분산 트랜잭션, 복제, 파티셔닝, 합의 알고리즘 같은 문제는 보통 백엔드 쪽이 더 깊고 어렵다.
- 하지만 그렇다고 프론트가 단순한 것은 아니다.
- 대규모 프론트는 캐시 무효화, 버전 스큐(사용자마다 서로 다른 버전의 프론트와 API를 보게 되는 꼬임), 점진 배포, 저사양 기기 대응, hydration budget(브라우저가 감당할 수 있는 초기 JS 실행량 관리), retry storm 제어(재시도가 한꺼번에 몰리는 현상 제어), edge personalization(CDN 가까운 위치에서 사용자별 응답을 다르게 만드는 처리), RUM 관측(실사용자 환경에서 성능을 수집하는 방식) 같은 별도의 복잡도를 가진다.
- 그래서 `프론트에는 서버처럼 복잡한 게 없다`는 판단은 틀렸다. 복잡도의 종류가 다를 뿐이다.

### 두 번째 배치에서 바로잡은 오해 3. `웹 앱과 모바일 앱`은 전체 경우의 수가 아니다

- 책은 이 문장에서 `현재 그림에서 대표적인 최종 사용자 단말` 두 개를 먼저 보여 준다.
- 그래서 `웹 앱과 모바일 앱만 웹 서버에 요청할 수 있다`는 뜻으로 읽으면 틀린다.
- HTTP 요청은 브라우저, 모바일 앱뿐 아니라 다른 서버, 크롤러, 웹훅 소비자(외부 서비스가 보낸 알림 요청을 받는 서버), 배치 작업, 모니터링 헬스체크(서버가 살아 있는지 확인하는 요청), CLI 도구(터미널 명령 프로그램)도 보낼 수 있다.
- 공식 문서도 HTTP는 브라우저와 웹 서버 사이뿐 아니라 프로그램과 프로그램 사이 통신에도 쓰인다고 설명한다.

짧은 예시:

- 백엔드 서버가 결제사 API를 호출한다.
- GitHub webhook(외부 서비스가 보내는 이벤트 알림 요청)을 받는 서버가 다시 내부 API를 호출한다.
- 모니터링 시스템이 `/health`를 주기적으로 찍는다.
- 검색 엔진 크롤러가 페이지를 수집한다.

이 전부가 HTTP 요청일 수 있다.

그래서 이 문장을 읽을 때는 `전체 경우의 수`가 아니라 `입문용 그림에서 먼저 보여 주는 대표 클라이언트`라고 해석해야 한다.

### 두 번째 배치에서 바로잡은 오해 4. 이 페이지 안에서 `웹 앱`은 좁게도 넓게도 섞여 쓴다

- 앞문장의 `웹 앱`은 요청을 보내는 브라우저 기반 클라이언트에 가깝다.
- 그런데 바로 아래 bullet의 `웹 애플리케이션`은 서버 구현 언어와 클라이언트 구현 언어를 함께 설명하므로, 웹 서비스 전체를 넓게 뭉뚱그린 표현에 가깝다.
- 즉 한 페이지 안에서 같은 단어가 서로 다른 범위로 쓰이고 있다.

독해 팁:

- 먼저 `지금 저자가 말하는 대상이 요청 발신자인가, 아니면 시스템 전체 구현인가`를 묻는다.
- 요청 발신자를 말하면 `웹 앱`은 브라우저 쪽 클라이언트다.
- 구현 방식을 말하면 `웹 애플리케이션`은 서버와 클라이언트를 함께 가진 웹 서비스 전체일 수 있다.

이건 네 독해력이 이상해서 생긴 혼란만은 아니다.
저자가 용어를 느슨하게 바꿔 쓴 것도 분명히 원인이다.

또 하나 중요한 점:

- `웹은 HTML, 모바일은 JSON`처럼 배타적으로 읽으면 틀린다.
- 웹 프론트도 JSON API를 아주 많이 쓴다.
- 모바일도 경우에 따라 HTML을 받을 수 있다.
- JSON 예시는 `모바일이 많이 이렇게 쓴다`는 설명용이지, `모바일만 이렇게 쓴다`는 규칙이 아니다.

### 세 번째 배치에서 바로잡은 오해 5. 서버를 극장처럼 보는 비유는 절반만 맞다

네 질문 원문은 `learner/events.ndjson`의 `system-design-event-007`에 그대로 보존했다. 여기서는 사고 교정 포인트를 학습용으로 다시 정리한다.

먼저 냉정하게 판정하면, 네가 떠올린 `입장 인원이 정해진 극장` 비유는 나쁘지 않다. 오히려 좋은 출발이다. 서버에도 처리 가능한 용량이 있고, 들어오는 요청이 처리 속도보다 많아지면 대기열이 길어지고 응답 시간이 늘어난다. 이 점에서는 비유가 맞다.

하지만 그 비유를 끝까지 밀고 가면 위험하다. 서버는 좌석 수 하나로 제한되는 극장이 아니다. 더 정확한 비유는 `주문 카운터, 키오스크, 주방, 창고, 결제 단말이 한 방에 몰려 있는 작은 식당`이다.

네가 추가로 말한 `문이 좁다`, `차선이 줄어드는 교통 병목` 비유는 특정 상황에는 맞다. 네트워크 대역폭, connection accept, 로드 밸런서 앞단처럼 입구 자체가 막히는 상황을 설명할 때는 좋은 비유다. 하지만 서버 전체를 이해하는 기본 비유로는 부족하다. 문 비유는 병목이 하나라고 가정한다. 실제 서비스는 로그인, 검색, 업로드, 결제, 재고 조회마다 막히는 지점이 달라진다.

- 카운터가 느리면 요청 접수가 밀린다.
- 주방이 느리면 처리 시간이 길어진다.
- 창고가 느리면 재고 조회가 막힌다.
- 결제 단말이 느리면 전체 회전율이 떨어진다.
- 손님 수가 같아도 주문 종류가 무거우면 병목 위치가 달라진다.

서버도 마찬가지다.

- CPU가 부족하면 계산이 느려진다.
- 메모리가 부족하면 GC, 스왑, OOM 같은 문제가 난다.
- 디스크 I/O가 부족하면 DB 읽기/쓰기가 느려진다.
- 네트워크 대역폭이 부족하면 응답 전송이 막힌다.
- DB connection pool이 부족하면 요청이 DB 앞에서 대기한다.
- 락 경합이 생기면 요청 수가 많지 않아도 특정 작업이 줄줄이 기다린다.

그러니까 네 비유는 `대기열이 생긴다`는 감각에는 맞지만, `무엇이 병목인지 계층별로 찾아야 한다`는 시스템 설계 감각에는 부족하다.

두 번째로, 네가 책의 전개에 대해 느낀 불편함도 그냥 틀린 감각이 아니다. 네 말처럼 이 문단은 `왜 사용자가 늘면 서버 하나로 부족한가`를 자세히 증명하지 않는다. 곧바로 `웹/모바일 트래픽 처리 서버와 데이터베이스 서버를 나누자`라는 해법으로 넘어간다.

이건 책이 매우 친절하게 설명한 흐름은 아니다. 하지만 시스템 설계 입문서에서는 꽤 흔한 흐름이다.

독해할 때는 이렇게 표시해라.

1. 주장: 사용자가 늘면 서버 하나로는 충분하지 않다.
2. 해법: 웹 계층과 데이터 계층을 분리한다.
3. 해법의 장점: 각각 독립적으로 확장할 수 있다.

즉 여기서 책은 `왜 부족한가`를 깊게 설명하는 문단이 아니라, `부족해졌다고 보고 첫 번째로 어떻게 나눌 것인가`를 보여 주는 문단에 가깝다. 네가 이 차이를 감지한 건 좋은 독해다. 다만 `적절치 못하다`에서 멈추기보다, `이 문장은 원인 설명인가, 해법 제시인가, 해법의 이점 설명인가`로 역할을 태깅하는 쪽이 더 발전적인 독해다.

네가 마지막에 정리한 `이렇게 세부적으로 나눠야 확장해야 될 부분을 포커스해서 확장할 수 있으니 의미 없이 모든 걸 확장할 필요가 없어 자원을 아낄 수 있구나`는 맞다. 하지만 여기서 끝내면 부족하다.

더 정확히는 이렇게 확장해야 한다.

- 비용: 병목인 계층만 키울 수 있으니 불필요한 자원 증설을 줄인다.
- 장애 격리: 웹 서버가 무거워져도 DB를 같은 프로세스 공간에서 같이 죽이지 않도록 경계를 만든다.
- 보안 경계: 브라우저나 모바일 앱이 DB에 직접 접근하지 않고 API 서버를 거치게 한다.
- 운영 책임 분리: 웹 서버 배포와 DB 백업/복구/튜닝을 다른 방식으로 관리할 수 있다.
- 확장 방식 분리: 웹 계층은 여러 대로 늘리기 쉽지만, 데이터 계층은 상태가 있으므로 복제, 샤딩, 일관성 문제를 따로 다뤄야 한다.

여기서 특히 조심할 점이 있다. `각각 독립적으로 확장할 수 있다`를 `둘 다 똑같이 쉽게 확장할 수 있다`로 읽으면 틀린다. 웹 계층은 무상태에 가깝게 만들면 수평 확장, 즉 서버를 여러 대 늘리는 방식이 비교적 쉽다. 반면 데이터베이스는 상태를 가진 계층이다. 데이터를 여러 곳에 복제하거나 나누면 읽기/쓰기 일관성, 장애 복구, 동기화 지연, 샤딩 키 같은 문제가 바로 따라온다.

프론트엔드 관점에서는 그림 1-3에서 이 점을 봐야 한다.

- 브라우저와 모바일 앱은 데이터베이스에 직접 붙지 않는다.
- 프론트는 웹/API 서버에 요청한다.
- 웹/API 서버가 인증, 권한, 입력 검증, 비즈니스 로직을 거쳐 DB에 read/write/update를 보낸다.
- DB는 데이터를 웹/API 서버에 반환하고, 웹/API 서버가 다시 프론트에 응답한다.

즉 프론트 화면에서 `데이터가 늦게 뜬다`는 현상은 단순히 React 렌더링 문제가 아닐 수 있다. API 서버 처리, DB query, DB connection pool, 인덱스, 캐시 미스, 네트워크가 모두 얽힌 결과일 수 있다.

이번 독해에서 네가 가져가야 할 사고 도구는 두 가지다.

1. 비유는 출발점이지 결론이 아니다.
2. 기술 문단은 `주장`, `이유`, `해법`, `효과`, `한계` 중 어떤 역할인지 표시하며 읽는다.

이 두 가지를 계속 연습하면 단순 개념 암기가 아니라 문장을 시스템 구조로 바꾸는 독해력이 올라간다.

추가로, `데이터 모델, 질의 패턴, 일관성 요구, 확장 방식, 운영 난이도`는 이렇게 풀어라.

- 데이터 모델: 데이터를 DB 안에서 어떤 단위, 필드, 관계, 중복 구조로 저장할지다. 도메인 이름만도 아니고 최종 JSON 모양만도 아니다. 예를 들어 주문을 `orders`, `users`, `products` 테이블로 나눌지, 주문 문서 안에 배송지와 상품 요약을 함께 넣을지, `USER#1`과 `ORDER#9` 같은 키로 묶어 둘지의 문제다.
- 질의 패턴: 앱이 자주 묻는 질문의 모양이다. `사용자별 최근 주문 20개`, `상품 상세와 리뷰 상위 3개`, `재고 차감`, `친구의 친구 추천`처럼 조건, 빈도, 읽기/쓰기 양을 포함한다.
- 일관성 요구: 같은 사실이 여러 곳에 있을 때 얼마나 빨리 같은 값이어야 하는지다. 재고 수량은 즉시 맞아야 하지만 추천 상품 목록은 조금 오래돼도 될 수 있다.
- 확장 방식: 데이터와 요청을 어떤 키와 파티션으로 나눠 트래픽을 분산할 수 있는지다. 한 사용자에게만 요청이 몰리는 hot key가 생기면 확장이 어려워진다.
- 운영 난이도: 백업, 복구, 장애 대응, 인덱스 관리, 샤딩 키 선택, 마이그레이션, 중복 데이터 보정, 모니터링까지 포함한 실제 유지 비용이다. `NoSQL이 항상 더 어렵다`가 아니라, 선택한 모델이 해결해 주는 문제와 새로 만드는 문제가 다르다는 뜻이다.

`API 응답 모양과 화면 상태 설계는 DB 모델의 영향을 받는다`는 문장도 정확히는 이렇게 읽어야 한다. 백엔드는 최종 JSON을 약속한 모양으로 내려줄 수 있다. 그래서 프론트가 같은 DTO를 받는다는 생각은 절반은 맞다. 하지만 같은 JSON을 만들 수 있다는 것과 같은 비용으로 만들 수 있다는 것은 다르다.

- RDBMS: 정규화된 테이블을 join해서 `ProductDetailDTO`를 만들 수 있다. 프론트가 받는 JSON은 문서형 DB와 같을 수도 있다. 차이는 `그 JSON을 만들기 위해` 백엔드가 여러 테이블을 join하거나, 경우에 따라 여러 서비스 응답을 조합해야 한다는 점이다. 그래서 join 비용, 인덱스, N+1, 캐시 전략이 중요해진다.
- 문서형 DB: 상품 문서 안에 판매자명, 리뷰 일부, 썸네일을 함께 넣으면 같은 `GET /products/:id`라도 한 번 읽어서 바로 내려주기 쉬워진다. 여기서 `API가 단순하다`는 말은 프론트가 받는 JSON 모양이 본질적으로 다르다는 뜻이 아니라, 화면 하나를 만들기 위한 백엔드 조합 단계나 프론트 호출 수가 줄 수 있다는 뜻이다. 대신 판매자명이 바뀌었는데 일부 문서만 갱신되면, 최신성 관리 실패가 프론트에서는 오래된 이름이나 요약이 보이는 stale UI로 드러날 수 있다.
- DynamoDB식 접근 패턴 모델: `상품 상세`, `사용자별 주문`, `카테고리별 인기 상품`처럼 미리 정한 질문은 빠르게 만들 수 있다. 대신 나중에 임의 필터링과 정렬이 늘어나면 새 인덱스, 새 테이블, 검색 엔진이 필요해질 수 있다.
- BFF/API Gateway: 백엔드가 여러 서비스나 저장소에서 데이터를 모아 프론트가 쓰기 좋은 모양으로 바꿔 줄 수 있다. 이 덕분에 프론트 호출 수는 줄어들지만, 그 변환 계층 자체가 운영해야 할 시스템이 된다.

프론트엔드에서는 이것이 이렇게 보인다.

- 한 API로 화면에 필요한 데이터가 다 오느냐, 여러 API를 조합해야 하느냐
- cursor pagination을 써야 하느냐, offset pagination도 버틸 수 있느냐
- optimistic UI를 써도 안전하냐, 서버 확정 후에만 보여줘야 하느냐
- stale 표시나 재검증 로직이 필요하냐
- partial update 후 캐시를 어느 범위까지 무효화해야 하느냐

따라서 DB가 프론트에 영향을 준다는 말은 `프론트가 SQL을 알아야 한다`는 뜻이 아니다. 같은 화면을 만들더라도 API의 비용, 응답 지연, 캐시 전략, stale 가능성, 상태 업데이트 방식이 DB 모델의 영향을 받는다는 뜻이다.

### 네 번째 배치에서 바로잡은 오해 6. NoSQL은 관계가 없어서 조인이 없는 게 아니다

네 질문 원문은 `learner/events.ndjson`의 `system-design-event-008`에 그대로 보존했다. 여기서는 사고 교정 포인트를 학습용으로 다시 정리한다.

먼저 냉정하게 말하면, `비-관계형 데이터베이스는 일반적으로 조인 연산을 지원하지 않는다`를 보고 이유를 기대한 것은 좋은 독해 반응이다. 책은 이 지점에서 충분히 친절하지 않다. 하지만 네가 잠정적으로 내린 `나눠져 있는 애들이 관계성이 없기 때문에 굳이 조인시킬 필요가 없다`는 결론은 틀렸다.

구분해야 할 것이 세 개다.

1. 데이터 사이의 관계: 주문과 사용자, 게시글과 댓글, 사람과 친구처럼 현실 데이터가 서로 연결되어 있다는 뜻이다.
2. 관계형 모델: 데이터를 테이블, 행, 열, 키 관계로 표현하고 SQL로 조합하는 방식이다.
3. 조인 연산: 여러 테이블이나 데이터 집합을 관련 키 기준으로 합쳐 하나의 결과처럼 조회하는 연산이다.

NoSQL은 1번, 즉 데이터 사이의 관계가 없다는 뜻이 아니다. NoSQL은 보통 2번과 3번, 즉 관계형 테이블 모델과 범용 SQL join을 중심에 두지 않는다는 뜻에 가깝다.

예를 들어 주문 상세 화면을 생각해 보자.

- RDBMS 방식: `orders`, `users`, `order_items`, `products`를 정규화해 두고 필요할 때 join으로 합친다.
- 문서형 방식: 주문 문서 안에 주문 당시의 사용자 이름, 배송지, 상품 요약을 일부 함께 넣어 둔다.
- 키-값/와이드 컬럼 방식: `customer#123`의 최근 주문, `order#456`의 상세처럼 접근 패턴에 맞춘 키로 읽는다.
- 그래프 방식: 사람, 게시글, 태그, 팔로우 관계를 노드와 관계선으로 두고 경로를 탐색한다.

여기서 중요한 점은 `관계가 있느냐`가 아니라 `그 관계를 매 요청마다 SQL join으로 계산할 것이냐, 미리 함께 저장할 것이냐, 참조를 따라갈 것이냐, 그래프 경로로 찾을 것이냐`다.

Neo4j 질문도 여기서 풀린다. Neo4j에는 관계가 있다. 오히려 관계를 아주 중요하게 다룬다. 하지만 그것은 RDBMS의 테이블 join 중심 관계가 아니라, 노드와 관계를 따라가는 graph pattern matching(그래프 모양을 찾아가는 질의)이다. 따라서 `그래프도 관계가 있는데 왜 비관계형인가?`라는 질문은 좋지만, 답은 `관계가 없어서 비관계형이 아니라, 관계형 테이블 모델이 아니어서 비관계형이다`다.

독해 훈련 포인트:

1. 용어가 부정형일 때는 `무엇이 없다는 뜻인가`를 좁혀라.
2. `비관계형`을 `관계 없음`으로 직역하지 말고 `관계형 테이블 모델 중심이 아님`으로 읽어라.
3. 책이 이유를 생략하면 다음 문장만 기다리지 말고, 앞에서 제시한 분류가 어떤 설계 철학을 암시하는지 되짚어라.

### 네 번째 배치에서 바로잡은 오해 7. 스케일업의 단점은 정의에서 추론해야 한다

네가 `트래픽이 적을 때는 수직적 확장이 좋은 선택`이라는 문장에서 이유를 찾지 못한 것도 이해된다. 책은 여기서 `왜 그런가`를 충분히 풀지 않고 바로 단점으로 넘어간다.

하지만 똑똑한 독자는 이유를 이미 외워서 아는 사람이 아니다. 정의를 보자마자 제약을 검사하는 사람이다.

`수직적 규모 확장(scale up)`의 정의:
한 서버에 더 좋은 CPU, 더 많은 RAM 같은 자원을 추가한다.

이 정의에서 바로 뽑아야 할 함의는 세 가지다.

1. 장점: 서버 수가 늘지 않으니 구조가 단순하다.
2. 한계: 한 대의 서버에 자원을 무한히 붙일 수 없다.
3. 장애: 여전히 한 대라서 그 서버가 죽으면 대신 받을 실행 주체가 없다.

그래서 트래픽이 적을 때 수직 확장이 좋은 이유는 `엄청난 성능 향상` 때문만이 아니다. 더 큰 이유는 단순함이다. 로드 밸런서, 여러 서버 배포, 세션 공유, 데이터 복제, 장애 전환 같은 분산 시스템 문제를 아직 감수하지 않아도 된다.

반대로 트래픽이 많아지면 단순함의 장점보다 한계가 더 커진다.

- 한 서버의 CPU, 메모리, 디스크, 네트워크에는 물리적 상한이 있다.
- 한 서버가 처리하는 동안 요청이 쌓이면 지연 시간이 급격히 늘어난다.
- 서버 한 대가 장애 나면 트래픽을 넘길 다른 서버가 없다.

여기서 `왜 자동복구 방안이나 다중화 방안을 제시하지 않는다`는 문장을 헷갈려 한 것도 중요한 포인트다. 답은 단순하다. 수직 확장은 capacity(처리할 수 있는 양)를 키우는 방법이지 availability(장애가 있어도 계속 쓸 수 있는 정도)를 만드는 방법이 아니다.

용어를 풀면:

- capacity: 시스템이 감당할 수 있는 처리량이나 용량이다.
- availability: 장애가 일부 있어도 사용자가 서비스를 계속 쓸 수 있는 정도다.
- failover: 고장 난 리소스 대신 건강한 리소스로 역할이나 트래픽을 넘기는 것이다.
- redundancy: 하나가 죽어도 대신할 수 있도록 같은 역할의 자원을 둘 이상 두는 것이다.
- single point of failure: 그 하나가 죽으면 전체가 멈추는 단일 장애점이다.

따라서 `왜 방안이 없어?`의 정답은 `절대 만들 수 없다`가 아니다. 만들 수 있다. 다만 만들려면 두 번째 서버, 복제본, 로드 밸런서, 헬스체크, 트래픽 전환 같은 다른 구조를 추가해야 한다. 그 순간 이미 `한 대를 더 크게 만드는 수직 확장만의 해법`을 넘어선다.

독해 훈련 포인트:

1. 정의를 읽으면 바로 `이 방식이 줄이는 복잡도`와 `이 방식이 남기는 제약`을 찾는다.
2. 장점 문장이 짧으면 `무엇이 단순해지는가`를 운영 관점으로 풀어 본다.
3. 단점 문장을 읽으면 `용량 문제인가, 장애 문제인가, 운영 복잡도 문제인가`로 분류한다.

이번 문단을 읽을 때의 더 나은 머릿속 흐름은 이렇다.

`scale up은 한 대를 키운다. 그러면 서버가 여러 대가 아니니까 단순하다. 그래서 작은 트래픽에는 괜찮다. 하지만 한 대의 최대치는 유한하고, 한 대가 죽으면 받을 다른 리소스가 없다. 그래서 큰 트래픽과 높은 가용성에는 scale out과 redundancy가 필요해진다.`

### 다섯 번째 배치에서 바로잡은 오해 8. 부하 분산 집합은 로드밸런서가 요청을 보낼 후보 서버 묶음이다

네가 `부하 분산 집합에 추가한다는 게 로드밸런서를 통해 분산시켜 줄 서버의 모수를 크게 만든다는 뜻이냐`고 물은 것은 좋은 질문이다. 결론부터 말하면 대체로 맞다. 다만 `모수`처럼 수학적 숫자로만 읽으면 핵심이 조금 빠진다.

더 정확히는 `부하 분산 집합`은 로드밸런서 뒤에 등록된 웹 서버들의 묶음, 즉 backend pool이나 target group에 가깝다. 로드밸런서는 이 묶음 안의 서버들 중 건강한(healthy) 서버에게만 새 요청을 보낸다. 그래서 `또 하나의 웹 서버를 추가한다`는 말은 단순히 숫자를 늘린다는 뜻이기도 하지만, 운영 관점에서는 `로드밸런서가 트래픽을 보낼 수 있는 건강한 웹 서버 후보를 하나 더 등록한다`는 뜻이다.

`웹 계층의 가용성이 향상된다`는 말도 성능과 섞어 읽으면 안 된다.

- 성능: 응답 시간이 짧아지거나 처리량이 늘어나는 것
- 가용성: 일부 서버가 고장 나도 사용자가 서비스를 계속 쓸 수 있는 정도

예를 들어 웹 서버가 한 대뿐이면 그 한 대가 죽는 순간 웹 계층 전체가 요청을 못 받는다. 하지만 웹 서버가 세 대이고 로드밸런서가 그중 건강한 두 대로 계속 요청을 보낼 수 있으면, 전체 서비스 성능은 조금 떨어질 수 있어도 `완전히 멈추는 상황`은 피할 수 있다. 이게 가용성 향상이다.

독해 훈련 포인트:

1. `집합`, `그룹`, `풀` 같은 말이 나오면 추상어로 넘기지 말고 `누가 누구를 모아 둔 목록인가`로 바꿔 읽는다.
2. `가용성`은 `빠름`이 아니라 `살아 있음`에 가깝게 읽는다.
3. 로드밸런서가 있으면 `몇 대로 나누는가`뿐 아니라 `고장 난 서버를 제외할 수 있는가`까지 같이 본다.

### 다섯 번째 배치에서 바로잡은 오해 9. 데이터베이스 주-부 관계는 임의 별명이 아니라 복제 구성이다

`서버 사이에 주-부 관계를 설정한다`는 문장을 보고 `DB를 두 개 띄워 놓고 내가 임의로 하나는 주, 하나는 부라고 부르는 건가?`라고 생각한 것도 자연스럽다. 하지만 이건 이름표 수준의 문제가 아니다.

여기서 `서버 사이`는 보통 서로 다른 데이터베이스 인스턴스나 노드를 뜻한다. 물리 서버일 수도 있고, VM일 수도 있고, 컨테이너나 별도 프로세스일 수도 있다. 로컬에서도 PostgreSQL 컨테이너 두 개를 띄워 primary/standby 구성을 실습할 수 있다.

하지만 핵심은 `두 개를 띄웠다`가 아니라 `복제 관계를 설정했다`는 점이다.

- 주 서버(primary): 보통 쓰기 원본을 받는 서버
- 부 서버(standby/replica): 주 서버의 변경을 받아 적용하는 복제본 서버

이 관계는 데이터베이스 엔진이 제공하는 복제 기능으로 구성한다. 예를 들어 PostgreSQL에서는 primary가 WAL(변경 로그)을 보내고 standby가 그것을 받아 재생(replay)한다. 따라서 이것은 임의 별명이 아니라 `누가 원본 변경을 만들고, 누가 그것을 따라가며, 누가 승격(promote)될 수 있는가`까지 포함한 역할 차이다.

여기서 프론트엔드가 가져가야 할 포인트도 있다. 복제가 비동기라면 부 서버는 순간적으로 주 서버보다 조금 늦을 수 있다. 그러면 사용자는 방금 저장한 데이터를 다시 읽었는데 아직 예전 값이 보이는 read-after-write 문제를 겪을 수 있다. 즉 데이터베이스 다중화는 백엔드 인프라 이야기 같아 보여도, 실제로는 프론트의 stale UI, 재검증, optimistic UI 설계와도 연결된다.

### 다섯 번째 배치에서 바로잡은 오해 10. 로컬 개발용 DB와 운영 복제 토폴로지는 다른 문제다

`데이터를 지역적으로 떨어진 여러 장소에 다중화시켜 놓을 수 있다`는 문장을 보고 `그럼 로컬에서 DB를 띄우고, 나중에 스테이징/프로덕션 DB를 두는 흐름을 말하는 건가?`라고 연결한 것은 아주 흔한 오해다. 하지만 이 둘은 층위가 다르다.

- 로컬/스테이징/프로덕션: 환경 구분
- primary/standby/replica: 한 환경 안에서의 역할 구분

즉 `로컬 -> 스테이징 -> 프로덕션`은 개발과 배포 단계의 흐름이고, 책의 `다중화`는 운영 환경 안에서 서비스를 계속 살려 두기 위해 데이터베이스를 여러 인스턴스에 복제해 두는 구조다.

클라우드에서는 이것이 꼭 `개발자가 물리 서버를 직접 만든다`는 뜻도 아니다. 관리형 서비스가 대신 인스턴스를 만들고 복제와 장애 전환을 관리해 주는 경우가 많다. 예를 들어 AWS RDS에서는 다음처럼 읽는 것이 더 정확하다.

- 같은 리전 안의 다른 가용 영역(AZ)에 standby를 두는 Multi-AZ: 주 목적은 고가용성과 빠른 장애 전환
- 다른 리전에 read replica를 두는 cross-region 복제: 주 목적은 재해 복구나 특정 지역의 읽기 성능 개선

네가 짚은 `그런데 먼 리전은 느려지지 않나?`는 맞는 걱정이다. 그래서 cross-region 복제는 무조건 기본값이 아니라, 지연과 복제 lag를 감수해도 얻는 이득이 있을 때만 쓴다. 대부분의 기본 고가용성 설계는 먼저 `같은 리전 안 다른 AZ`부터 생각한다.

독해 훈련 포인트:

1. `개발 환경 이야기인가, 운영 배치 이야기인가`를 먼저 분리한다.
2. `지역적으로 떨어진 여러 장소`를 보면 곧바로 `다른 리전`으로 단정하지 말고, 같은 리전 안 다른 AZ부터 떠올린다.
3. 복제는 `데이터가 여러 군데 있다`에서 끝나지 않고, `왜 거기 두는가`, `얼마나 늦게 따라오는가`, `장애 시 누가 승격되는가`까지 함께 본다.

### 여섯 번째 배치에서 바로잡은 오해 11. 캐시는 위치마다 목적과 조정 지점이 다르다

이번 질문은 앞으로 기록 방식까지 바꾸는 기준점이다. 질문 원문은 `learner/events.ndjson`의 `system-design-event-013`에 줄바꿈과 목록을 보존해 남겼고, 화면에서는 `질문 원문`, `질문 다듬기`, `왜 막혔나`, `정리`로 나눠 보여 준다.

먼저 질문 자체를 더 좋게 바꾸면 이렇게 된다.

1. 이 문맥의 캐시는 브라우저 캐시가 아니라 웹 서버와 데이터베이스 사이에 두는 서버 측 캐시 계층인가?
2. 데이터베이스 호출 비용은 네트워크 왕복보다 항상 큰가, 아니면 네트워크 왕복과 DB 내부 처리, 락, 커넥션 대기, 직렬화까지 합친 총비용이 커질 수 있다는 뜻인가?
3. 내 서비스의 액세스 패턴은 어떤 지표와 사용자 행동으로 파악하며, 그에 따라 TTL, key 설계, invalidation, eviction 정책을 어떻게 정해야 하는가?
4. Redis/Memcached 같은 서버 캐시와 TanStack Query 같은 브라우저 서버 상태 캐시는 같은 `캐시`라는 말을 쓰지만, 조정 지점과 면접 답변 방식이 어떻게 다른가?

여기서 중요한 교정은 `DB 호출 비용이 네트워크보다 항상 큰가?`라고 묻는 방식이다. 이 질문은 좋은 출발이지만, 너무 이분법적이다. 더 정확한 질문은 `DB 호출의 총비용은 어떤 구성요소로 이뤄지고, 내 시스템에서는 무엇이 실제 병목인지 어떻게 측정하는가?`다.

DB 호출 비용에는 네트워크 왕복만 있는 것이 아니다.

- 애플리케이션 서버에서 DB까지의 네트워크 왕복
- connection pool에서 연결을 기다리는 시간
- SQL 파싱과 실행 계획 선택
- 인덱스 탐색
- 디스크 또는 메모리 접근
- 락 경합과 트랜잭션 대기
- 결과 직렬화와 역직렬화
- 애플리케이션 서버에서 JSON 응답으로 가공하는 비용

그래서 `DB가 네트워크보다 무조건 비싸다`는 말은 틀리다. 같은 가용 영역 안의 단순 key lookup은 매우 빠를 수 있고, 먼 리전 네트워크 왕복이 더 큰 경우도 있다. 하지만 대규모 웹 서비스에서는 많은 요청이 같은 데이터를 반복해서 읽고, DB가 공유 병목이 되기 쉽기 때문에 책은 DB 호출을 대표 예시로 든다.

이것을 감으로 판단하면 안 된다. 실제로는 아래 지표로 확인해야 한다.

- API p95/p99 latency: 느린 사용자 요청의 꼬리 지연
- DB query time: 쿼리 자체가 걸린 시간
- slow query log: 느린 쿼리 목록
- connection pool wait: DB 연결을 얻기 위해 기다린 시간
- cache hit ratio: 캐시가 대신 처리한 비율
- DB CPU, memory, disk I/O: DB 서버 내부 병목
- tracing/APM: 한 요청이 웹 서버, 캐시, DB 중 어디에서 시간을 쓰는지 보여 주는 추적

이 페이지의 그림에서 말하는 캐시는 브라우저 캐시가 아니다. 웹 서버와 데이터베이스 사이에 놓는 서버 측 캐시 계층이다. 다만 캐시는 위치마다 다르게 존재한다.

- 브라우저 HTTP 캐시: 사용자 기기 안에서 정적 응답이나 HTTP 응답을 재사용한다.
- TanStack Query: 브라우저 메모리 안에서 서버 상태를 query key 기준으로 재사용하고 refetch를 제어한다.
- CDN 캐시: 사용자 가까운 엣지 서버에서 JS, CSS, 이미지, 비디오 같은 정적 콘텐츠를 재사용한다.
- 서버 측 캐시: Redis/Memcached처럼 웹 서버와 DB 사이에서 DB 읽기 부하를 줄인다.
- DB 내부 버퍼 캐시: DB 엔진이 자주 읽는 페이지를 메모리에 올려 둔다.

`액세스 패턴`은 단어 뜻처럼 그냥 접근한다는 의미가 아니다. 실무에서는 데이터가 실제로 어떻게 쓰이는지를 뜻한다.

- 어떤 데이터가 자주 읽히는가?
- 어떤 데이터가 자주 바뀌는가?
- 읽기가 많은가, 쓰기가 많은가?
- 같은 key에 요청이 몰리는 hot key가 있는가?
- 사용자가 방금 쓴 값을 바로 다시 봐야 하는가?
- 조금 오래된 값을 보여줘도 괜찮은가?
- 데이터 크기가 큰가, 작은가?
- 사용자별 데이터인가, 전체 공통 데이터인가?

예를 들어 쇼핑몰이라면 이렇게 나뉜다.

- 카테고리 목록: 읽기 많고 변경 적음. 긴 TTL이나 CDN/서버 캐시가 잘 맞는다.
- 상품 상세: 읽기 많지만 가격과 판매 상태는 주의해야 한다. 기본 정보와 재고를 분리해 캐시할 수 있다.
- 재고: stale하면 주문 실패나 과판매가 생긴다. 캐시를 짧게 잡거나 쓰기 경로에서 강하게 무효화해야 한다.
- 사용자 장바구니: 사용자별 데이터이고 쓰기가 잦다. 공용 서버 캐시보다 사용자별 서버 상태 캐시나 짧은 TTL이 필요할 수 있다.
- 공지사항: 변경은 드물고 모두가 읽는다. 길게 캐시하기 좋다.

`캐시 메모리를 과할당한다`는 문장은 주로 Redis/Memcached 같은 서버 캐시 용량 설계에 직접 해당한다. 서버 캐시는 인스턴스 크기, shard 수, `maxmemory`, eviction policy를 운영자가 정한다. Redis 공식 문서가 설명하는 것처럼 메모리 한계에 도달하면 LRU, LFU 같은 정책으로 어떤 key를 내보낼지 결정한다.

TanStack Query는 다르다. 개발자가 브라우저 메모리를 `몇 GB`로 잡지는 않는다. 대신 다음을 조정한다.

- `staleTime`: 캐시된 데이터를 fresh로 볼 시간
- `gcTime`: 더 이상 쓰이지 않는 inactive query cache를 메모리에서 치우기 전까지의 시간
- query key 설계: 같은 데이터를 같은 key로 재사용할 수 있는가
- invalidation: mutation 후 어떤 query를 무효화할 것인가
- infinite query의 page 보관 수: 무한 스크롤에서 몇 페이지를 계속 들고 있을 것인가
- persister 사용 여부: 브라우저 저장소에 캐시를 유지할 것인가

면접에서 `캐싱 전략은 어떻게 잡으셨나요?`라고 물으면 전략 이름을 줄줄 외우는 답변은 약하다. 좋은 답변은 아래 순서로 간다.

1. 무엇을 캐시했는가?
2. 왜 캐시했는가?
3. 어디에 캐시했는가?
4. key는 어떻게 설계했는가?
5. TTL과 invalidation은 어떻게 정했는가?
6. stale해도 되는 데이터와 안 되는 데이터를 어떻게 나눴는가?
7. cache miss와 cache 장애 시 fallback은 무엇인가?
8. hit ratio, p95 latency, DB 부하 같은 지표로 효과를 어떻게 확인했는가?

예시 답변:
`상품 카테고리와 공지사항은 변경이 드물고 읽기가 많아서 서버 캐시와 CDN 캐시를 길게 잡았습니다. 반면 재고와 결제 상태는 stale하면 사용자 피해가 커서 길게 캐시하지 않았고, mutation 이후 관련 query를 무효화했습니다. 프론트에서는 TanStack Query의 staleTime으로 탭 이동 시 불필요한 refetch를 줄이고, 장바구니 변경 후에는 관련 query를 invalidate했습니다. 효과는 API p95 latency와 DB read QPS, cache hit ratio로 확인했습니다.`

이 답변은 방향은 맞지만 아직 너무 교과서적이다. 실무형으로 바꾸려면 캐시 위치를 먼저 나누고, 각 위치마다 `무엇/왜/어디/key/TTL/무효화/stale/fallback/지표`를 따로 말해야 한다. 아래 예시는 Google web.dev, Meta Relay 문서와 GitHub, TanStack Query 문서와 GitHub, Netflix EVCache 문서와 GitHub를 확인해 만든 실전형 답변 틀이다. 숫자가 명시된 부분은 출처가 공개한 숫자만 사용하고, 공개 지표가 없는 부분은 면접에서 측정해야 할 지표로만 둔다. 숫자를 지어내면 안 된다.

#### 캐싱 전략 면접 답변 예시 A. 프론트 정적 자산 HTTP 캐시

Google의 `stale-while-revalidate` 사례처럼, 프론트 캐시는 단순히 `이미지는 CDN에 올렸습니다`에서 끝나지 않는다. 실제 답변은 이렇게 말할 수 있다.

`사용자가 매번 받는 정적 스크립트와 이미지 중 변경 빈도가 낮고 rollback 위험이 낮은 파일을 브라우저 HTTP 캐시와 CDN에 캐시했습니다. 목적은 origin 요청과 다운로드 대기 시간을 줄이는 것이었습니다. key는 URL과 파일명 hash를 기준으로 잡았고, 내용이 바뀌면 hash가 바뀌도록 했습니다. 오래된 파일이 치명적이지 않은 자산은 Cache-Control에 max-age와 stale-while-revalidate를 함께 두어, 사용자는 즉시 캐시된 파일을 쓰고 브라우저가 뒤에서 새 버전을 확인하게 했습니다. 다만 API 응답이나 권한별 HTML처럼 사용자별로 달라지는 데이터는 같은 방식으로 길게 캐시하지 않았습니다. 효과는 cache hit ratio, transferred bytes, LCP, script load time, origin request 감소로 봤고, 배포 직후 문제를 줄이기 위해 versioned asset과 rollback 가능성을 같이 확인했습니다.`

여기서 중요한 점은 `stale-while-revalidate`가 프론트에도 진짜 실무 전략이라는 것이다. Google 사례에서는 GPT 광고 스크립트에 짧은 `max-age`와 더 긴 `stale-while-revalidate` 창을 적용해, 캐시된 스크립트를 즉시 쓰고 백그라운드에서 갱신하도록 했다. 공개 사례는 수십억 요청 샘플에서 early script load와 광고 노출/수익 지표가 개선됐다고 보고한다. 이 예시는 면접에서 `TTL을 왜 그렇게 잡았나?`에 답하기 좋다. 최신 파일을 매번 강제하면 안전해 보이지만 로딩이 느려지고, 너무 길게 잡으면 rollback과 버전 꼬임이 위험해진다. 그래서 `빠른 로딩을 위해 stale을 어느 정도 허용하되, 파일명 hash와 배포 정책으로 위험을 줄였다`라고 설명해야 한다. [SRC-SD-044]

#### 캐싱 전략 면접 답변 예시 B. 프론트 서버 상태 캐시

Meta의 Relay와 TanStack Query 같은 클라이언트 데이터 라이브러리는 `프론트에서도 캐싱 전략이 있다`는 가장 좋은 예시다. 여기서 캐시는 CDN이나 Redis가 아니라 브라우저 메모리 안의 서버 상태 캐시다.

`상품 목록, 상품 상세, 사용자 프로필처럼 여러 화면에서 반복해서 읽는 서버 상태는 프론트의 query cache에 저장했습니다. 목적은 라우트 이동, 탭 전환, 컴포넌트 remount 때 같은 데이터를 다시 받아오느라 빈 화면과 중복 요청이 생기는 것을 줄이는 것이었습니다. key는 resource 이름과 id, filter, sort, page cursor를 포함해 설계했습니다. 예를 들어 product detail은 ['product', productId], 검색 결과는 ['products', { keyword, sort, cursor }]처럼 구분했습니다. staleTime은 데이터 성격에 맞춰 다르게 잡았습니다. 카테고리나 공지처럼 잘 안 바뀌는 데이터는 길게 두고, 재고·알림 수·장바구니처럼 사용자 행동 직후 바뀌는 데이터는 mutation 후 invalidate하거나 optimistic update를 적용했습니다. fallback은 캐시가 있으면 stale 값을 먼저 보여 주고 background refetch를 돌리되, 결제·재고처럼 틀리면 안 되는 화면은 submit 전에 서버 재검증을 하도록 했습니다. 효과는 화면 이동 시 중복 request 수, skeleton 노출 시간, API p95, query cache hit, mutation 후 stale UI 발생률로 확인했습니다.`

Relay 문서에서는 같은 데이터를 다시 방문할 때 store에 있는 데이터를 재사용하고, 부족하거나 오래된 데이터는 fetch policy에 따라 네트워크로 보충하는 흐름을 설명한다. Relay Environment의 Store는 normalized data cache로 소개된다. 이것은 프론트 캐시가 `그냥 메모리에 저장`이 아니라, 같은 엔티티가 목록과 상세에 동시에 등장할 때 한 레코드로 연결해 업데이트 일관성을 유지하려는 설계라는 뜻이다. TanStack Query는 normalized cache보다는 query key 단위 캐시에 가깝지만, `staleTime`, `gcTime`, `invalidateQueries`, optimistic update 같은 질문은 동일하게 나온다. [SRC-SD-039] [SRC-SD-040] [SRC-SD-045] [SRC-SD-046] [SRC-SD-047] [SRC-SD-048] [SRC-SD-049]

냉정하게 말하면, 프론트 캐시 답변에서 `TanStack Query 썼습니다`만 말하면 약하다. 라이브러리 이름은 전략이 아니다. 면접관이 듣고 싶은 것은 `왜 그 데이터는 stale을 허용했고, 왜 그 데이터는 mutation 후 즉시 무효화했으며, key가 틀리면 어떤 버그가 생기고, 실패했을 때 사용자는 무엇을 보게 되는가`다.

#### 캐싱 전략 면접 답변 예시 C. 서버 측 DB 캐시

서버 측 캐시는 면접에서 Redis/Memcached를 말할 때 가장 자주 나오는 형태다. Netflix EVCache처럼 대규모 서비스에서는 캐시가 단순 보조 저장소가 아니라 지역성, 복제, fallback, 짧은 TTL, 클라이언트 재시도 정책까지 포함한 운영 대상이 된다.

`읽기 요청이 많고 계산 비용이 큰 추천 결과, 상품 상세의 일부 파생 데이터, 공개 설정값을 Redis/Memcached 계층에 캐시했습니다. 목적은 DB read QPS와 계산 비용을 줄이고 p95 latency를 낮추는 것이었습니다. key는 도메인과 버전, 식별자, 파라미터를 포함했습니다. 예를 들어 product:v2:{productId}:summary처럼 버전을 넣어 스키마 변경 시 기존 key와 충돌하지 않게 했습니다. TTL은 데이터 변경 빈도와 stale 허용 범위로 나눴습니다. 공지·카테고리는 길게, 재고·가격은 짧게 또는 캐시하지 않게 했습니다. write가 발생하면 관련 key를 삭제하거나 이벤트로 invalidation했습니다. cache miss는 DB를 읽고 캐시에 채우는 cache-aside로 처리했고, 캐시 장애 시에는 짧은 timeout 후 DB fallback을 하되 트래픽 폭주를 막기 위해 request coalescing이나 rate limit을 함께 두었습니다. 효과는 cache hit ratio, DB read QPS, p95/p99 latency, eviction rate, Redis CPU/memory, timeout/error rate로 확인했습니다.`

Netflix EVCache 문서는 짧은 TTL, 지역적으로 가까운 캐시 읽기, 실패 시 다른 복제본 fallback, 일시적 불일치 가능성을 전제로 한 설계를 설명한다. 이 사례가 중요한 이유는 `캐시는 빠르게 하기 위한 것`에서 끝나지 않고, `캐시가 장애 나면 어디로 우회할지`, `복제본끼리 값이 잠깐 달라도 되는지`, `어느 지역의 캐시를 먼저 읽을지`까지 캐싱 전략에 포함된다는 점을 보여 주기 때문이다. [SRC-SD-050] [SRC-SD-051] [SRC-SD-052]

면접에서 가장 안전한 결론은 이것이다. `캐싱 전략은 cache-aside냐 read-through냐를 외우는 문제가 아니라, 데이터가 얼마나 자주 읽히고 쓰이는지, stale이 사용자 피해로 이어지는지, 캐시가 실패했을 때 느려질 뿐인지 서비스가 죽는지, 그리고 그 판단을 어떤 지표로 검증했는지 말하는 문제다.`

#### 이 예시에서 쓰인 용어 풀이

이 섹션은 앞으로 예시 바로 아래에 반드시 붙인다. 용어를 모르면 답변을 외워도 실전에서 다시 무너진다. 특히 약어는 처음 볼 때 뜻을 풀고, 그 다음 문맥에서 어떻게 쓰이는지 연결해야 한다.

- TTL: Time To Live의 줄임말. 캐시에 저장된 값을 얼마나 오래 유효하다고 볼지 정한 시간이다. 예를 들어 TTL 5분이면 5분 동안은 캐시 값을 믿고, 이후에는 다시 가져오거나 갱신 대상으로 본다.
- key 또는 cache key: 캐시에 값을 저장하고 다시 찾기 위한 이름이다. `product:v2:123:summary`처럼 도메인, 버전, 식별자, 파라미터를 넣어야 서로 다른 데이터를 잘못 공유하지 않는다.
- origin: CDN이나 브라우저 캐시가 최종적으로 원본 파일을 가져오는 원래 서버 또는 원본 저장소다. CDN cache hit이면 origin까지 가지 않고, cache miss이면 origin에 요청이 간다.
- Cache-Control: HTTP 응답 헤더 중 하나다. 브라우저나 CDN이 이 응답을 캐시해도 되는지, 얼마나 오래 보관해도 되는지 알려준다.
- max-age: `Cache-Control` 안에서 쓰는 지시어다. `max-age=60`이면 응답을 60초 동안 fresh한 캐시로 보겠다는 뜻이다.
- stale: 캐시에 있긴 하지만 최신이라고 확신하기 어려운 오래된 상태다. stale이 항상 나쁜 것은 아니다. 공지 목록처럼 조금 늦게 바뀌어도 괜찮은 데이터는 stale을 잠깐 허용할 수 있다.
- stale-while-revalidate: 오래된 캐시를 일단 즉시 보여 주고, 뒤에서 새 데이터를 다시 가져와 다음 요청부터 갱신된 값을 쓰게 하는 HTTP 캐시 전략이다. 사용자는 빠르게 보지만, 아주 최신값은 잠시 늦게 반영될 수 있다.
- rollback: 새 배포나 변경이 문제를 만들었을 때 이전 버전으로 되돌리는 일이다. 정적 파일을 너무 길게 캐시하면 rollback해도 사용자의 브라우저가 오래된 파일을 계속 들고 있을 수 있다.
- 파일명 hash: 파일 내용으로 만든 짧은 식별자를 파일명에 붙이는 방식이다. 예를 들어 `app.a1b2c3.js`처럼 만들면 내용이 바뀔 때 파일명도 바뀌므로 긴 캐시와 새 배포를 함께 쓰기 쉬워진다.
- versioned asset: 버전이 붙은 정적 자산이다. 파일명 hash가 대표적인 방식이고, 브라우저나 CDN이 예전 파일과 새 파일을 구분하게 해 준다.
- cache hit ratio: 전체 요청 중 캐시가 처리한 비율이다. 100번 요청 중 80번을 캐시에서 처리했다면 hit ratio는 80%다. 캐시를 넣은 효과를 볼 때 가장 먼저 확인하는 지표 중 하나다.
- transferred bytes: 실제 네트워크로 내려받은 데이터 양이다. 캐시가 잘 작동하면 같은 파일을 다시 다운로드하지 않으므로 이 값이 줄어든다.
- LCP: Largest Contentful Paint의 줄임말. 사용자가 보는 화면에서 큰 콘텐츠가 표시되기까지 걸린 시간이다. 이미지, 큰 텍스트, hero 영역이 늦게 뜨면 LCP가 나빠진다.
- server state: 서버가 원본으로 관리하는 데이터의 프론트 표현이다. 상품 목록, 사용자 프로필, 장바구니처럼 브라우저가 직접 소유하지 않고 서버에서 받아오는 상태를 말한다.
- query key: TanStack Query 같은 라이브러리에서 서버 상태 캐시를 구분하는 키다. key가 너무 뭉뚱그려지면 다른 필터의 검색 결과가 섞이고, 너무 잘게 쪼개지면 재사용이 줄어든다.
- invalidate 또는 invalidation: 캐시된 값을 더 이상 믿지 않겠다고 표시하거나 삭제하는 일이다. 예를 들어 장바구니 수량을 바꾼 뒤 `cart` query를 invalidate하면 다음에는 새 값을 다시 가져온다.
- optimistic update: 서버 응답을 기다리기 전에 성공할 것이라고 가정하고 UI를 먼저 바꾸는 방식이다. 좋아요 버튼을 누르자마자 숫자를 올렸다가 실패하면 되돌리는 식이다.
- fallback: 기본 경로가 실패했을 때 쓰는 대체 경로다. 캐시 서버가 죽으면 DB를 직접 읽거나, stale 캐시를 임시로 보여 주거나, 일부 기능을 숨기는 방식이 fallback이다.
- normalized data cache: 같은 엔티티를 여러 화면에서 중복 저장하지 않고 하나의 레코드처럼 관리하는 프론트 데이터 캐시 방식이다. 예를 들어 상품 목록과 상품 상세가 같은 상품 `id=123`을 가리키면 한 곳의 변경이 여러 화면에 반영되기 쉬워진다.
- fetch policy: 캐시를 먼저 볼지, 네트워크를 먼저 볼지, 캐시가 없을 때만 네트워크를 갈지 정하는 규칙이다. Relay의 `store-or-network` 같은 설정이 여기에 해당한다.
- background refetch: 화면에는 기존 캐시 데이터를 보여 주면서 뒤에서 새 데이터를 다시 요청하는 방식이다. 사용자는 빈 화면을 덜 보고, 데이터는 점진적으로 갱신된다.
- p95와 p99 latency: 요청 지연 시간을 빠른 순서대로 줄 세웠을 때 95번째 백분위, 99번째 백분위에 해당하는 값이다. 평균보다 느린 사용자 경험을 더 잘 보여 준다.
- QPS: Queries Per Second의 줄임말. 초당 처리되는 요청 또는 질의 수다. DB read QPS가 줄었다는 말은 DB가 받는 읽기 요청 수가 줄었다는 뜻이다.
- eviction: 캐시 공간이 부족할 때 기존 값을 내보내는 일이다. 자주 쓰는 값이 계속 eviction되면 cache miss가 늘고 성능이 떨어진다.
- eviction rate: 일정 시간 동안 캐시에서 값이 얼마나 자주 밀려나는지 나타내는 지표다. 이 값이 높으면 캐시 용량, TTL, key 설계, 데이터 크기를 다시 봐야 한다.
- request coalescing: 같은 데이터에 대한 cache miss가 동시에 많이 발생했을 때, DB 요청을 여러 번 보내지 않고 하나만 보내도록 합치는 방식이다. 캐시가 비었을 때 갑자기 DB가 두들겨 맞는 상황을 줄인다.
- rate limit: 일정 시간 동안 허용할 요청 수를 제한하는 장치다. 캐시 장애나 miss 폭증 때 DB나 외부 API가 무너지는 것을 막는 안전장치로 함께 쓴다.

독해 훈련 포인트:

1. `캐시`라는 단어를 보면 먼저 `어느 위치의 캐시인가`를 묻는다.
2. `비용이 크다`는 문장을 보면 `무엇과 비교해서, 어떤 지표로 확인할 것인가`를 묻는다.
3. `액세스 패턴` 같은 추상어는 `읽기/쓰기 빈도, hot key, 최신성, 데이터 크기, 사용자 범위`로 쪼갠다.
4. 면접 답변은 전략 이름이 아니라 도메인, trade-off, 지표로 구성한다.

### 일곱 번째 배치에서 바로잡은 오해 12. CDN은 파일 저장소 하나가 아니라 edge와 origin의 배달 구조다

네 질문 원문은 `learner/events.ndjson`의 `system-design-event-014`에 줄바꿈과 목록을 보존해 남겼다. 이번 질문은 CDN을 `파일을 S3에 올리고 CloudFront로 배포한다`는 실무 동작으로는 알고 있지만, 그 말 안에 들어 있는 `edge server`, `origin`, `cache key`, `invalidation`, `object versioning`을 분리하지 못해 생긴 막힘이다.

먼저 질문 자체를 더 좋게 바꾸면 이렇게 된다.

1. CloudFront 같은 CDN에서 `edge server`, `origin`, `distribution`, `cache key`는 각각 어떤 역할을 하며, S3는 그중 어디에 놓이는가?
2. `동적 콘텐츠 캐싱`에서 동적이라는 말은 JavaScript가 화면을 바꾼다는 뜻인가, 아니면 요청 path/query/cookie/header에 따라 서버 응답 bytes가 달라질 수 있다는 뜻인가?
3. `클라이언트 fallback`은 브라우저 코드, 모바일 앱, 서버, DNS/라우팅 중 어느 계층에서 구현 가능한가?
4. CDN 캐시를 바꾸는 방법은 `무효화 API로 기존 캐시를 지우는 방식`과 `URL/파일명을 바꿔 새 object로 서비스하는 방식`으로 어떻게 나뉘는가?

`서버의 네트워크`라는 표현은 네트워크 요청 한 번을 말하는 게 아니다. CloudFront 같은 CDN 사업자가 여러 지역의 edge location 또는 POP에 서버를 두고, DNS/라우팅/캐시 정책으로 사용자를 가까운 서버에 연결하는 전체 인프라 묶음을 뜻한다. React 빌드 파일을 S3에 올린 행위 자체가 `서버의 네트워크 행위`인 것은 아니다. 더 정확히는 `S3 bucket을 origin으로 두고, CloudFront distribution이 edge network를 통해 그 object를 사용자 가까이에서 제공하도록 구성했다`고 말해야 한다. [SRC-SD-053] [SRC-SD-054] [SRC-SD-058]

흐름은 이렇게 읽으면 된다.

1. 브라우저가 `https://...cloudfront.net/app.abc123.js`를 요청한다.
2. CloudFront는 DNS/라우팅을 통해 사용자와 가까운 edge location으로 요청을 보낸다.
3. edge cache에 파일이 있으면 바로 내려준다.
4. 없으면 origin인 S3 bucket 또는 HTTP 서버에 가서 파일을 가져온다.
5. edge는 응답의 캐시 정책을 보고 TTL 동안 저장하고, 다음 가까운 사용자는 edge에서 받는다.

`동적 콘텐츠 캐싱`에서 동적이라는 말은 JavaScript가 DOM을 바꾼다는 뜻이 아니다. 같은 URL의 HTML이라도 로그인 사용자, 쿠키, 언어, 기기, A/B 테스트, query string, request header에 따라 서버가 다른 HTML bytes를 만들 수 있으면 동적이다. CDN은 이런 값을 cache key에 포함해 서로 다른 응답을 따로 캐시할 수 있다. 그래서 `HTML은 정적이고 JS가 동적이다`라는 문장은 너무 거칠다. 더 좋은 판별 기준은 `이 응답이 모든 사용자에게 같은 bytes인가, 아니면 요청 문맥에 따라 달라지는가?`다. [SRC-SD-055] [SRC-SD-059]

S3는 `온라인 저장소`라는 말 때문에 막연해지기 쉽다. S3는 서버 프로세스를 띄워 요청마다 HTML을 계산하는 웹 서버가 아니라, bucket 안에 object를 저장하고 HTTP/API로 꺼내 쓰게 하는 객체 저장소다. 여기서 object는 파일 bytes와 metadata를 합친 저장 단위다. 프론트에서 `이미지 업로드`, `정적 빌드 파일 배포`, `다운로드 링크`, `백업/로그 저장`에 자주 쓰는 이유가 이것이다. CloudFront를 붙이면 S3는 원본 저장소, 즉 origin이 되고 CloudFront edge가 사용자 가까이에서 파일을 대신 내려준다. [SRC-SD-058]

CDN 장애 문맥의 `클라이언트`는 요청하는 쪽을 넓게 가리킨다. 브라우저나 모바일 앱일 수도 있고, 어떤 구조에서는 서버일 수도 있다. 프론트 정적 자산에서는 브라우저가 CDN URL을 요청하는 경우가 많다. 다만 아주 냉정하게 말하면, JS 번들 자체가 CDN에서 실패하면 React 코드가 실행되기 전일 수 있으므로 모든 fallback을 앱 내부에서 해결할 수는 없다. 가능한 대처는 여러 계층에 나뉜다.

- HTML에서 `<script onerror>`로 보조 origin URL을 시도한다.
- Service Worker가 캐시된 이전 자산을 제공한다.
- 운영자가 DNS, CDN 설정, origin routing을 바꿔 우회한다.
- 서버가 CDN URL 대신 origin URL을 내려주도록 설정을 바꾼다.
- 다중 CDN 또는 origin fallback을 CDN/로드밸런싱 계층에서 구성한다.

`CDN 서비스 사업자가 제공하는 API`로 제거한다는 말은 네가 이해한 대로 CloudFront invalidation 같은 기능이다. 아직 TTL이 남은 edge cache를 지워서 다음 요청이 origin의 새 파일을 보게 만든다. 다만 매 배포마다 무조건 invalidation에 의존하면 비용과 전파 시간이 생긴다. 그래서 정적 자산에서는 가능하면 versioned file name을 쓰는 것이 흔하다. [SRC-SD-056] [SRC-SD-057]

`오브젝트 버저닝`은 이 문맥에서 S3 bucket versioning만을 말하는 것이 아니라, CDN이 보는 object URL을 바꿔 새 콘텐츠로 보이게 만드는 cache busting 패턴이다. 책의 `image.png?v=2`가 그 예다. React/Vite/Next 빌드 파일의 `app.abc123.js` 같은 content hash도 같은 계열로 보면 된다. 파일 내용이 바뀌면 URL도 바뀌므로, CDN과 브라우저는 기존 캐시와 새 파일을 다른 object로 본다.

프론트가 신경 쓸 실무 포인트는 다음과 같다.

- `app.[contenthash].js`, `style.[contenthash].css` 같은 정적 자산은 긴 `Cache-Control`과 immutable cache가 잘 맞는다.
- `index.html`은 최신 JS/CSS 해시 파일명을 가리키는 진입점이므로 짧게 캐시하거나 revalidate하게 두는 경우가 많다.
- 같은 URL의 파일 내용을 덮어쓰면 사용자의 HTML과 JS 버전이 꼬일 수 있다.
- 긴급 교체나 잘못 올라간 파일 삭제에는 CloudFront invalidation을 쓴다.
- 정적 자산 CDN cache key에는 불필요한 cookie/header를 넣지 않는다. 넣으면 같은 파일인데도 사용자마다 다른 캐시로 쪼개져 hit ratio가 떨어진다.
- 사용자별 HTML이나 API 응답은 무심코 전역 CDN 캐시에 길게 넣으면 개인정보 노출 또는 stale UI 위험이 생긴다.

#### 이번 배치 용어 풀이

- CDN network: 여러 지역의 edge server, DNS/라우팅, 캐시 저장소, origin 연결 규칙이 함께 동작하는 인프라다. 서버 한 대도 아니고 네트워크 요청 한 번도 아니다.
- edge server: 사용자 가까운 곳에서 파일을 내려주는 CDN 서버다. CloudFront에서는 edge location 또는 POP 맥락으로 읽으면 된다.
- origin: CDN이 cache miss 때 원본 파일을 가져오는 서버나 저장소다. S3 bucket, 웹 서버, 로드밸런서가 origin이 될 수 있다.
- Amazon S3: AWS의 객체 저장소 서비스다. 파일을 object로 저장하고 bucket이라는 그릇 안에서 HTTP/API로 꺼내 쓰게 한다.
- bucket: S3에서 object를 담는 최상위 컨테이너다.
- object: S3/CDN 문맥에서 파일 bytes와 metadata를 가진 저장 단위다.
- cache key: CDN이 `이 요청은 기존 캐시와 같은가?`를 판단하는 기준이다. path, query string, cookie, header가 포함될 수 있다.
- query string: URL의 `?v=2&lang=ko` 같은 부분이다. CDN cache key에 포함하면 버전이나 조건별로 캐시가 나뉜다.
- request header: 브라우저나 클라이언트가 HTTP 요청에 붙여 보내는 부가 정보다. 언어, 인증, user-agent 같은 값이 들어갈 수 있다.
- cookie: 브라우저가 특정 사이트에 대해 저장했다가 요청마다 보낼 수 있는 작은 데이터다. 사용자별 응답을 만드는 데 쓰이면 CDN 캐시에서 특히 조심해야 한다.
- TTL: Time To Live의 줄임말이다. CDN이나 캐시가 응답을 얼마나 오래 유효하다고 볼지 정한 시간이다.
- invalidation: 아직 TTL이 남은 CDN 캐시를 강제로 제거하는 작업이다. CloudFront invalidation이 대표 예다.
- object versioning: URL이나 파일명을 바꿔 CDN이 다른 object로 보게 하는 방식이다. `image.png?v=2`와 `app.abc123.js`가 여기에 속한다.
- content hash: 파일 내용으로 만든 해시를 파일명에 붙이는 방식이다. 내용이 바뀌면 파일명도 바뀌어 긴 캐시를 안전하게 쓰기 좋다.
- fallback: CDN 장애나 cache miss 폭증처럼 기본 경로가 실패했을 때 쓰는 대체 경로다.
- static asset: 빌드된 JS, CSS, 이미지, 폰트처럼 배포 전까지 내용이 거의 바뀌지 않고 모든 사용자에게 같은 bytes로 내려가는 파일이다.

독해 훈련 포인트:

1. `정적/동적`을 파일 확장자로 판단하지 말고, 모든 사용자에게 같은 bytes인지로 판단한다.
2. `서버의 네트워크` 같은 표현은 요청 행위인지, 인프라 묶음인지 먼저 분리한다.
3. `클라이언트`라는 단어는 브라우저만 뜻한다고 단정하지 말고, 요청 주체와 fallback 구현 계층을 나눠 읽는다.
4. `버전`이라는 말은 S3 bucket versioning, URL versioning, 파일명 hash가 섞일 수 있으므로 문맥을 확인한다.

### 여덟 번째 배치에서 바로잡은 오해 13. 무상태는 상태가 없다는 뜻이 아니라 웹 서버에 묶지 않는다는 뜻이다

이번 입력은 한 메시지 안에 세 개의 발췌문이 있었으므로, `system-design-event-015`, `system-design-event-016`, `system-design-event-017`로 나눠 저장했다. 앞으로도 이런 형식은 메시지 전체가 아니라 발췌문 단위로 나눠 다룬다.

#### 발췌 1. 상태 정보를 웹 계층에서 제거한다

네 질문은 `세션 데이터가 정확히 무엇인지`, `웹 계층에서 제거한다는 말이 브라우저 상태를 말하는지`, `DB에서 필요한 데이터를 가져오는 프론트 방식과 비슷한지`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. 여기서 `세션 데이터`는 로그인 세션만 뜻하는가, 아니면 요청 사이에 이어져야 하는 사용자별 상태 전체를 뜻하는가?
2. `웹 계층에서 상태를 제거한다`는 말은 브라우저 상태를 없앤다는 뜻인가, 로드밸런서 뒤의 개별 웹 서버 메모리/디스크에 사용자 상태를 두지 않는다는 뜻인가?
3. 상태를 외부 저장소에 둔다는 것은 DB/Redis/NoSQL/session store/token 방식 중 어떤 선택지를 포함하며, 각각 어떤 trade-off가 있는가?

냉정하게 말하면, 네 질문의 좋은 점은 `세션 데이터`를 로그인 방식 중 하나로만 좁혀도 되는지 의심했다는 것이다. 이건 아주 좋은 감각이다. 다만 `상태`를 React state나 프론트 하드코딩 데이터와 같은 층위로 섞으면 안 된다.

여기서 세션 데이터는 로그인 세션이 대표 예지만 그것만 뜻하지 않는다. 사용자의 로그인 여부, user id, 권한, 장바구니, 임시 작업 상태처럼 여러 HTTP 요청 사이에 이어져야 하는 사용자별 서버 상태를 말한다. `상태 정보를 웹 계층에서 제거한다`는 말은 브라우저 상태를 없애라는 뜻이 아니라, 로드밸런서 뒤의 개별 웹 서버 메모리나 로컬 디스크에 사용자 상태를 저장하지 말라는 뜻이다.

예를 들어 서버 1 메모리에만 `userId=42는 로그인됨`이라는 상태가 있으면 서버 2는 같은 사용자의 요청을 받아도 그 사실을 모른다. 그래서 상태를 Redis, 관계형 DB, NoSQL, 별도 session store, 또는 서명된 토큰 같은 공유 가능한 위치로 빼야 한다.

네가 든 `프론트에서 하드코딩하지 않고 API 응답으로 필요 데이터를 가져온다`는 비유는 방향감은 맞지만 정확한 대응은 아니다. 프론트 비유로 말하면 `데이터를 컴포넌트 파일 안에 박아 두지 않고 외부 source of truth에서 읽는다`에 가깝다. 하지만 이 페이지의 핵심은 프론트 코드가 아니라 `웹 서버 A만 아는 상태 때문에 웹 서버 B가 요청을 처리하지 못하는 문제`다.

#### 발췌 2. 같은 클라이언트 요청은 항상 같은 서버로 가야 한다

네 질문은 `이건 서버가 여러 대일 때의 문제이고, 단일 서버라면 상관 없는 문제 아닌가?`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. `같은 클라이언트는 같은 서버로 가야 한다`는 제약은 여러 웹 서버가 있고 상태가 각 서버 로컬에 있을 때만 생기는가?
2. 단일 서버에서는 sticky session 문제가 없더라도, 단일 서버 자체가 병목·단일 장애점·세션 유실 위험을 갖는다는 점은 어떻게 구분해야 하는가?

판정하면, 네 해석은 절반은 맞다. 단일 서버라면 모든 요청이 그 한 대로 가므로 `어느 서버로 보낼지`를 고르는 문제는 없다. sticky session 문제도 없다. 하지만 그래서 단일 서버가 상태를 로컬에 둬도 안전하다는 뜻은 아니다. 서버 한 대가 병목이고, 죽으면 전체가 멈추며, 세션을 메모리에 두었다면 재시작 때 세션이 날아갈 수 있다.

여러 서버로 늘리는 순간에는 문제가 달라진다. 사용자 A의 세션이 서버 1에만 있으면 사용자 A 요청은 계속 서버 1로 가야 한다. 이때 로드밸런서가 같은 사용자를 같은 서버에 묶는 기능이 sticky session 또는 session affinity다. AWS Application Load Balancer도 target group 수준에서 sticky session을 제공한다. 하지만 이 기능은 서버 추가, 제거, rolling deploy, 장애 복구, scale-in을 복잡하게 만든다. [SRC-SD-062]

#### 발췌 3. 자동 규모 확장은 웹 서버를 자동으로 추가하거나 삭제한다

네 질문은 `AWS에서 말하는 그 오토스케일링이 맞는지`, `무상태가 아니면 오토스케일링은 못하는지`, `프론트에는 어떤 영향이 있는지`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. 책의 `autoscaling`은 일반 개념이고, AWS에서 EC2 웹 서버에 대응되는 서비스 이름은 `Amazon EC2 Auto Scaling`과 `Auto Scaling group`으로 봐도 되는가?
2. 상태 의존 웹 서버에서도 autoscaling 자체는 가능하지만, 왜 세션 유실·sticky session·scale-in·draining 문제 때문에 위험하거나 복잡해지는가?
3. 프론트엔드는 autoscaling과 rolling deploy 상황에서 인증, 업로드, WebSocket, 재시도, optimistic update를 어떻게 설계해야 사용자 경험이 깨지지 않는가?

책의 autoscaling은 일반 개념이다. AWS에서 EC2 기반 웹 서버를 자동으로 늘리고 줄이는 전형적인 이름은 `Amazon EC2 Auto Scaling`과 그 안의 `Auto Scaling group`이다. Auto Scaling group은 최소/희망/최대 인스턴스 수를 기준으로 EC2 인스턴스 집합을 관리하고, 정책에 따라 인스턴스를 시작하거나 종료한다. ALB 뒤에 Target Group을 두고 Auto Scaling Group이 웹 서버 인스턴스를 늘리거나 줄이는 구성이 흔하다. [SRC-SD-060] [SRC-SD-061]

`무상태가 아니면 오토스케일링을 못한다`는 결론은 너무 강하다. 정확히는 `할 수는 있지만 복잡하고 위험해진다`다. 새로 추가된 서버는 기존 로컬 세션을 모르고, scale-in으로 종료되는 서버에 붙어 있던 사용자는 세션을 잃을 수 있으며, sticky session을 쓰면 로드밸런서와 배포/장애 복구가 복잡해진다. 그래서 상태를 외부 저장소로 빼는 것이 자동 확장의 기본 전제처럼 다뤄진다.

프론트에 보이는 실무 증상은 꽤 구체적이다.

- 사용자가 배포 중 갑자기 로그아웃된다.
- 장바구니나 결제 단계가 서버에 따라 사라졌다 나타난다.
- 업로드 진행 상태가 scale-in이나 재시작 때 끊긴다.
- WebSocket 또는 SSE 연결이 끊긴 뒤 재연결이 어색하다.
- mutation 재시도로 결제, 좋아요, 쿠폰 사용 같은 동작이 중복 처리된다.
- optimistic update는 성공했다고 보였는데 서버 상태가 다른 서버로 가며 엇갈린다.

그래서 프론트도 특정 서버에 붙어 있다는 가정을 하지 않아야 한다. 인증 실패를 401/419로 구분해 처리하고, 안전한 요청만 재시도하며, 결제나 주문 같은 mutation은 idempotency key를 써야 하고, 업로드는 재개 가능한 프로토콜을 고려해야 한다. 실시간 연결은 끊김과 재연결을 정상 흐름으로 설계해야 한다.

#### 이번 배치 용어 풀이

- 상태 정보: 요청 하나만 보고는 알 수 없고 여러 요청 사이에 이어져야 하는 사용자별 또는 업무별 정보다. 로그인 여부, 권한, 장바구니, 임시 작업 상태 등이 포함된다.
- 세션 데이터: 사용자의 연속된 이용 맥락을 식별하고 유지하기 위한 데이터다. 로그인 세션이 대표적이지만 반드시 로그인만 뜻하지 않는다.
- 무상태 웹 계층: 개별 웹 서버가 사용자별 상태를 직접 들고 있지 않은 웹 서버 계층이다. 상태가 아예 없다는 뜻이 아니다.
- 상태 정보 의존적인 아키텍처: 특정 사용자의 요청이 특정 서버의 로컬 상태에 의존하는 구조다.
- sticky session: 로드밸런서가 같은 사용자의 요청을 같은 서버로 계속 보내는 기능이다.
- session affinity: sticky session과 비슷하게 같은 세션을 같은 target에 묶으려는 성질이다.
- shared storage: 여러 웹 서버가 공통으로 접근하는 외부 저장소다. Redis, DB, NoSQL 등이 될 수 있다.
- autoscaling: 트래픽이나 정책에 따라 서버 수를 자동으로 조절하는 기능이다.
- Auto Scaling group: AWS EC2 Auto Scaling에서 EC2 인스턴스를 자동 확장 대상으로 묶은 그룹이다.
- scale-in: 트래픽이 줄었을 때 서버 수를 줄이는 작업이다.
- draining: 종료할 서버에 새 요청을 보내지 않고 기존 요청이 끝나길 기다리는 절차다.

독해 훈련 포인트:

1. `상태`라는 단어를 보면 React state인지, 서버 로컬 상태인지, DB에 저장된 상태인지 먼저 구분한다.
2. `무상태`는 상태가 없다는 뜻이 아니라 요청 처리 서버가 상태를 직접 소유하지 않는다는 뜻으로 읽는다.
3. `가능/불가능`으로 결론내기 전에 `가능하지만 어떤 운영 비용과 장애 모드가 생기는가`를 묻는다.
4. 프론트에서 보이는 랜덤 로그아웃, 장바구니 초기화, 업로드 중단은 서버 상태 배치 문제일 수 있다.

### 아홉 번째 배치에서 바로잡은 오해 14. 데이터 센터는 한 URL 뒤에 숨은 여러 실제 경로다

이번 입력은 `데이터 센터` p33~35와 `메시지 큐` 도입부가 함께 들어왔다. 화면에서는 목차 절 기준으로 `1장 · 33~35p 데이터 센터`와 `1장 · 35p 메시지 큐 도입`을 분리했고, 사용자가 요청한 대로 메시지 전체를 하나로 묶지 않고 발췌문 단위로 `system-design-event-018`부터 `system-design-event-024`까지 나눠 저장했다.

그림은 더 이상 세로 텍스트 목록으로 길게 펼치지 않는다. 원문 이미지는 저장하지 않되, 리더 화면에서는 `도식 자료` 요약 카드와 `그림에 적힌 원문 레이블 보기` 접힘 영역으로 보존한다. 그림 자체를 완전히 재현할 수 없는 경우에는 `어떤 그림이 있었는지`, `그 그림이 어떤 역할을 했는지`, `레이블 원문`을 분리해 둔다.

#### 발췌 1. 가장 가까운 데이터 센터와 geoDNS

네 질문은 `어떤 원리로 가장 가까운 데이터 센터로 안내되는가`, `이런 세부 원리를 바로 궁금해하는 태도가 독해 속도를 떨어뜨리는가`, `문해력과 사고력을 올리려면 어떻게 교정해야 하는가`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. geoDNS는 사용자의 실제 GPS 위치가 아니라 어떤 네트워크 단서로 가까운 데이터 센터 IP를 선택하는가?
2. 책을 읽을 때 `지금 모르면 다음 문장이 안 읽히는 질문`과 `원리는 궁금하지만 잠시 보류해도 되는 질문`을 어떻게 구분해야 하는가?
3. 세부 원리를 궁금해하는 태도를 버리지 않으면서도 독해 속도를 유지하려면 어떤 질문 주차장과 시간 제한 전략을 써야 하는가?

가장 가까운 데이터 센터는 보통 GPS처럼 정밀한 물리 위치 하나로 결정되지 않는다. DNS resolver 위치, 클라이언트 subnet 정보가 전달되는 경우의 네트워크 단서, 지역 정책, latency data, health check 결과를 조합한다. AWS Route 53에도 geolocation routing, latency-based routing, failover routing이 따로 있다. [SRC-SD-063] [SRC-SD-064] [SRC-SD-065]

냉정하게 말하면, 네 질문 습관은 장점과 단점이 둘 다 있다. 장점은 단어를 대충 넘기지 않는다는 점이다. 단점은 매 문장에서 내부 구현까지 즉시 파고들면 작업 기억이 금방 꽉 차서 문단의 큰 방향을 놓칠 수 있다는 점이다. 인지부하 이론은 학습 중 불필요하게 많은 세부 요소가 작업 기억을 차지하면 스키마 형성이 어려워진다고 설명한다. 반대로 self-explanation 연구는 자기 질문과 자기 설명이 이해를 높일 수 있음을 보여 준다. 그래서 결론은 `질문하지 마라`가 아니라 `질문을 분류하라`다. [SRC-LEARN-006] [SRC-LEARN-007]

사용할 규칙은 세 가지다.

1. Blocking question: 지금 모르면 다음 문장이 안 읽히는 질문은 즉시 푼다.
2. Mechanism question: 원리는 궁금하지만 지금 흐름에는 없어도 되는 질문은 질문 주차장에 둔다.
3. Design question: 설계 선택과 trade-off에 영향을 주는 질문은 별도 기록으로 남겨 뒤에서 다시 푼다.

운전 중 차선 변경이 느리다는 자기 관찰도 같은 방식으로 볼 수 있다. 매번 모든 변수를 낱개로 계산하면 느리다. 대신 `후방 차 거리`, `상대 속도`, `빈 공간`, `내 이동 경로`를 묶음 패턴으로 보는 훈련이 필요하다. 독해도 `이 문장의 역할`, `다음 문장 예고`, `지금 필요한 질문`을 묶어 보는 훈련이 필요하다.

#### 발췌 2. 심각한 장애와 장애 감지

네 질문은 `심각한 장애가 무엇인지`, `그 장애를 프론트에서 API 에러코드처럼 알 수 있는지`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. 데이터 센터 단위 failover를 유발하는 장애는 서버 한 대 장애와 어떻게 다른가?
2. 장애 감지는 사용자 브라우저가 API 에러를 받은 뒤 판단하는가, 아니면 health check, synthetic monitoring, metrics, global load balancer가 먼저 판단하는가?
3. 프론트가 직접 할 수 있는 장애 대응과 인프라 계층이 하는 트래픽 전환은 어떻게 구분해야 하는가?

심각한 장애는 서버 한 대가 500을 내는 정도만 뜻하지 않는다. 데이터 센터 전력 장애, 네트워크 단절, 로드밸런서 장애, 웹 계층 대량 실패, DB 장애, 캐시 장애, 잘못된 배포, 특정 리전 클라우드 서비스 장애, 높은 packet loss와 latency가 포함될 수 있다.

장애 감지도 프론트가 에러 코드를 본 뒤에야 시작되는 것이 아니다. 보통 health check, synthetic monitoring, 실제 사용자 모니터링, 로그/메트릭/트레이싱, SLO 지표를 보고 라우팅 계층이 unhealthy 리소스를 제외하거나 다른 데이터 센터로 트래픽을 돌린다. 프론트에서 5xx나 timeout을 보는 것은 사용자 경험을 보호하기 위한 신호이지, 글로벌 failover의 유일한 근거가 아니다.

프론트가 할 일은 트래픽을 전 세계적으로 바꾸는 것이 아니라, 장애가 사용자에게 덜 날카롭게 드러나게 하는 것이다. 예를 들면 재시도/backoff, offline 안내, stale data 표시, read-only 모드, error boundary, optimistic update rollback 같은 UX가 여기에 들어간다.

#### 발췌 3. 여러 위치에서 테스트한다는 말

네 질문은 `웹사이트는 하나인데 여러 위치에서 테스트한다는 게 무엇인지`, `직접 이동해야 하는지`, `접속만 보면 되는지`, `데이터 동기화까지 봐야 하는지`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. `여러 위치에서 테스트`는 물리적으로 사람이 이동하라는 뜻인가, 여러 리전/네트워크 위치의 테스트 에이전트로 같은 사용자 플로우를 검증하라는 뜻인가?
2. 다중 데이터 센터 테스트는 단순 접속 확인, latency 측정, DNS routing 확인, 데이터 동기화 확인, failover drill 중 무엇을 포함하는가?
3. 프론트 관점에서는 지역별 정적 자산 로딩, API schema, 로그인/결제 플로우, stale data 표시를 어떻게 확인해야 하는가?

여러 위치에서 테스트한다는 말은 개발자가 비행기 타고 돌아다니라는 뜻이 아니다. 여러 지역의 synthetic monitoring, VPN/프록시, 클라우드 테스트 에이전트, 브라우저 테스트 서비스, RUM을 써서 같은 사용자 플로우를 여러 위치에서 검증한다는 뜻이다.

테스트도 단순히 홈페이지가 열리는지만 보는 것이 아니다. DNS가 올바른 데이터 센터로 보내는지, CDN 정적 자산이 잘 내려오는지, API latency와 error rate가 어떤지, 로그인과 장바구니가 유지되는지, read-after-write가 깨지지 않는지, 장애 시 failover가 되는지, 각 데이터 센터가 같은 배포 버전을 쓰는지까지 본다.

#### 발췌 4. 자동화된 배포 도구의 역할

네 해석은 `자동화 덕분에 동일한 서비스가 마법처럼 생기는 게 아니라, 데이터 센터가 많아질수록 관리하기 어려워지는 일을 자동화해 신경 쓸 것을 줄여 주는 것`이었다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. 자동화된 배포 도구가 `동일한 서비스 설치`를 보장한다는 말은 마법처럼 같게 만든다는 뜻인가, 같은 artifact와 설정을 반복 가능하게 배포해 drift를 줄인다는 뜻인가?
2. 다중 데이터 센터에서 수동 배포가 왜 version mismatch, schema mismatch, configuration drift를 만들기 쉬운가?
3. 배포 자동화가 같게 만드는 대상은 코드, 컨테이너 이미지, 환경 변수, DB migration, feature flag 중 무엇인가?

네 해석은 맞다. 다만 `신경 쓸 것이 줄어든다`에서 끝내면 조금 약하다. 자동화의 핵심은 같은 배포 산출물, 같은 설정 템플릿, 같은 migration 절차, 같은 feature flag 상태를 재현 가능하고 감사 가능하게 적용하는 것이다.

프론트 실무에서는 `A 지역은 새 JS bundle, B 지역은 옛 API`, `한 데이터 센터만 환경 변수가 달라 CDN URL이 틀림`, `DB migration 순서가 달라 특정 리전에서만 화면이 깨짐` 같은 문제가 실제로 생길 수 있다. 자동화는 이런 configuration drift를 줄인다.

#### 발췌 5. 시스템 컴포넌트 분리와 독립 확장

네 질문은 `컴포넌트로 분리하라는 게 DB, 로드밸런서, 웹 서버 같은 덩어리 분리인지`, `독립 확장이 DB만 확장하거나 웹 서버만 확장하는 뜻인지`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. 여기서 컴포넌트는 UI 컴포넌트가 아니라 어떤 실행/운영 단위인가?
2. 웹 서버, DB, 캐시, CDN, 메시지 큐, worker를 분리하면 각각 어떤 병목을 따로 확장할 수 있는가?
3. 독립 확장은 단순히 서버 수를 늘리는 것뿐 아니라 책임과 장애 격리까지 포함하는가?

맞다. 여기서 컴포넌트는 React 컴포넌트가 아니라 시스템 컴포넌트다. 웹 서버, DB, 캐시, CDN, 메시지 큐, worker, 검색 서버, 객체 저장소처럼 배포되고 실행되고 운영되는 단위다. 독립적으로 확장한다는 말은 웹 트래픽이 병목이면 웹 서버를 늘리고, 읽기 DB가 병목이면 replica를 늘리고, 이미지 처리가 병목이면 worker를 늘리는 식으로 필요한 부분만 키운다는 뜻이다.

그리고 여기에는 단순 비용 절약만 있는 것이 아니다. 책임 분리, 장애 격리, 배포 단위 분리도 포함된다. 메시지 큐는 이 흐름에서 `즉시 처리하지 않아도 되는 일`을 웹 요청 처리에서 떼어 내기 위한 장치다.

#### 발췌 6. 메시지 큐의 무손실

네 질문은 `무손실이 어떤 원리인지`, `어떻게 보장하는지`, `왜 메시지 큐여야 하는지`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. 메시지 큐의 `무손실`은 어떤 저장 방식, ack, retry, replication으로 보장되는가?
2. 메시지가 절대 한 번만 처리된다는 뜻인가, 아니면 보통 at-least-once 전달이라 소비자 idempotency가 필요한가?
3. 메시지 큐는 producer와 consumer의 속도 차이, 일시 장애, 긴 작업을 어떻게 분리해 주는가?

메시지 큐의 무손실은 메시지를 디스크나 복제 저장소에 기록하고, 소비자가 처리 완료 ack를 보내기 전까지 삭제하지 않으며, 실패하면 재전달하거나 dead-letter queue로 보내는 방식으로 만든다. 하지만 이것은 `정확히 한 번 처리`와 다르다. Amazon SQS 표준 큐는 최소 한 번 전달을 제공하므로 중복 메시지가 생길 수 있다. 그래서 소비자는 idempotent하게 만들어야 한다. [SRC-SD-066]

메시지 큐가 필요한 이유는 웹 요청과 느린 작업의 속도 차이를 흡수하기 위해서다. 예를 들어 이미지 업로드 API는 `업로드 접수 완료`를 빨리 돌려주고, 썸네일 생성은 메시지 큐에 넣어 worker가 처리하게 할 수 있다. 그러면 웹 서버는 빠르게 응답하고, worker는 따로 늘릴 수 있다.

#### 발췌 7. 비동기 통신과 Kafka

네 질문은 `카프카도 컴포넌트인가`, `카프카가 메시지 큐로 알고 있는데 이 문맥에 들어가는가`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. Kafka는 전통적인 메시지 큐와 완전히 같은가, 아니면 이벤트 스트리밍 플랫폼이지만 메시징 컴포넌트 역할도 하는가?
2. 이 문맥의 컴포넌트는 제품 이름이 아니라 시스템 안에서 어떤 역할을 맡는 실행/운영 단위인가?
3. queue, log, stream, pub/sub는 어떤 점이 비슷하고 어떤 점이 다른가?

Kafka도 이 문맥에서는 컴포넌트라고 부를 수 있다. 다만 전통적인 FIFO 큐와 완전히 같지는 않다. Kafka는 분산 commit log 기반의 event streaming platform이고, producer가 topic에 record를 쓰고 consumer group이 offset을 관리하며 읽는다. 그래도 생산자와 소비자를 비동기적으로 분리하고, 메시지를 durable하게 저장하며, 소비자를 확장할 수 있다는 점에서 이 절의 메시지 큐 개념과 연결된다. [SRC-SD-067] [SRC-SD-068]

면접에서는 `Kafka는 그냥 메시지 큐입니다`라고 말하면 조금 뭉개진다. 더 정확한 답은 `Kafka는 메시지 큐처럼 producer-consumer를 decouple하는 데 쓸 수 있지만, 본질적으로는 partitioned replicated log 기반의 이벤트 스트리밍 플랫폼입니다`에 가깝다.

#### 이번 배치 용어 풀이

- 데이터 센터: 서버, 네트워크, 전력, 냉각 설비가 모여 서비스를 운영하는 장소 또는 클라우드 인프라 지역 단위다.
- geoDNS: 도메인을 항상 같은 IP로 바꾸지 않고, 사용자 위치나 DNS resolver 위치, latency/health 정책에 따라 다른 IP를 돌려주는 DNS 방식이다.
- authoritative DNS: 특정 도메인의 최종 DNS 답을 가진 권한 있는 DNS 서버다.
- health check: 서비스가 살아 있고 정상 처리 가능한지 주기적으로 확인하는 검사다.
- synthetic monitoring: 가짜 사용자가 여러 지역에서 정해진 시나리오를 반복 실행해 서비스 상태를 확인하는 모니터링이다.
- RUM: Real User Monitoring의 줄임말이다. 실제 사용자 브라우저나 앱에서 수집한 성능·에러 데이터를 보는 방식이다.
- deployment artifact: 빌드된 서버 이미지, 컨테이너 이미지, JS/CSS 번들처럼 배포 대상이 되는 고정된 결과물이다.
- configuration drift: 수동 수정과 누락으로 데이터 센터나 서버별 설정이 조금씩 달라지는 문제다.
- 메시지 큐: 생산자가 만든 작업 메시지를 보관하고 소비자가 나중에 꺼내 처리하게 하는 중간 저장소다.
- durability: 메시지나 데이터가 저장된 뒤 장애가 나도 쉽게 사라지지 않도록 보호하는 성질이다.
- ack: consumer가 메시지를 성공적으로 처리했음을 큐에 알리는 확인 신호다.
- at-least-once delivery: 메시지를 잃지 않기 위해 한 번 이상 전달하는 보장이다. 중복 처리가 생길 수 있다.
- idempotency: 같은 요청을 여러 번 처리해도 결과가 한 번 처리한 것과 같게 만드는 성질이다.
- dead-letter queue: 계속 실패한 메시지를 따로 모아 원인 분석과 재처리를 할 수 있게 하는 큐다.
- Kafka: 전통적인 큐와 완전히 같지는 않지만, producer/consumer와 durable log를 통해 비동기 메시징과 이벤트 스트리밍을 구현하는 분산 플랫폼이다.

독해 훈련 포인트:

1. `가까운` 같은 단어는 물리 거리인지, 네트워크 latency인지, 운영 정책인지 확인한다.
2. 모든 세부 원리를 즉시 파고들지 말고 `blocking`, `mechanism`, `design` 질문으로 분류한다.
3. `테스트`는 접속 확인만이 아니라 라우팅, 데이터, 배포 버전, 장애 전환을 포함할 수 있다.
4. `무손실`은 절대 마법 보장이 아니라 저장, ack, 재시도, 중복 처리 설계가 함께 만드는 성질이다.

### 1장 · 36p 메시지 큐 확장 예시

이번 배치는 메시지 큐가 producer와 consumer 사이의 결합을 낮추고, worker 수를 큐 길이에 맞춰 조정할 수 있게 만드는 부분이다.

#### 발췌 1. 메시지 큐와 느슨한 결합

네 질문은 `Zustand를 도입하면 props drilling과 의미 없는 props 통로 컴포넌트가 줄어드는 것처럼, 메시지 큐도 비슷한 경우라고 봐도 되는가`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. Zustand가 props drilling을 줄여 UI 컴포넌트 사이 결합을 낮추는 것과, 메시지 큐가 서버/서비스 사이 결합을 낮추는 것은 어떤 추상적 공통점이 있는가?
2. 둘 다 `중간 완충 계층`처럼 보이지만, 메시지 큐에는 비동기 처리, durability, retry, ack, worker scaling 같은 추가 의미가 있다는 점을 어떻게 구분해야 하는가?

비유는 절반은 좋고 절반은 위험하다. 좋은 점은 둘 다 직접 연결을 줄인다는 것이다. props drilling에서는 중간 컴포넌트가 의미 없는 전달자 역할을 하고, Zustand 같은 store는 필요한 컴포넌트가 상태를 직접 구독하게 해 UI 결합을 줄인다. 메시지 큐도 웹 서버가 worker를 직접 호출하지 않고 큐에 job을 넣어 생산자와 소비자의 속도, 생존 여부, 확장 단위를 분리한다.

하지만 같은 기술로 보면 안 된다. Zustand는 같은 브라우저 런타임 안의 상태 공유이고, 메시지 큐는 네트워크 너머의 서비스 사이에서 durable storage, ack, retry, dead-letter queue, queue depth 기반 worker scaling을 다룬다. 면접에서는 `비유로는 props drilling을 줄이는 store처럼 직접 의존을 줄인다고 볼 수 있지만, 메시지 큐는 비동기·내구성·장애 격리가 추가된다`라고 말하는 편이 정확하다.

#### 이 예시에서 쓰인 용어 풀이

- 느슨한 결합: 한 컴포넌트가 다른 컴포넌트의 위치, 속도, 생존 여부를 덜 알아도 동작할 수 있게 만드는 설계 성질이다.
- job: 큐에 넣는 작업 단위다. 예를 들어 이미지를 300x300 썸네일로 만드는 요청 하나가 job이 될 수 있다.
- worker: 큐에서 job을 꺼내 실제 처리를 수행하는 작업 서버나 백그라운드 프로세스다.
- queue depth: 큐에 쌓인 메시지 수다. worker를 늘릴지 줄일지 판단하는 대표 신호다.
- backpressure: 생산 속도가 소비 속도보다 빨라 큐나 버퍼가 쌓일 때 시스템이 속도를 제한하거나 압력을 흡수해야 하는 현상이다.

### 1장 · 36~38p 로그, 메트릭 그리고 자동화

이번 배치는 로그, 메트릭, 자동화를 책의 짧은 설명보다 실무 기준으로 더 세밀하게 분리했다. 특히 프론트 관점에서는 `로그`, `사용자 행동 분석`, `세션 리플레이`, `RUM`, `비즈니스 메트릭`을 한 바구니에 넣으면 안 된다.

#### 발췌 1. 에러 로그를 모니터링하는 것은 중요하다

네 질문은 `로그를 원문 그대로 남기는지`, `백엔드와 프론트의 로그는 어떻게 남기는지`, `사용자 액션 수집은 로그와 다른지`, `GA4와 Hotjar 같은 도구는 어디에 들어가는지`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. 운영 로그는 자유 텍스트 원문을 그대로 남기는가, 아니면 level, requestId, userIdHash, route, release, stack 같은 필드를 가진 structured log로 남겨야 하는가?
2. 백엔드 로그, 프론트 error monitoring, RUM, 제품 analytics, session replay는 목적과 보안 주의점이 어떻게 다른가?
3. 프론트 실무에서 Sentry, GA4/GTM, Web Vitals, Hotjar류 도구를 어떤 목적별 조합으로 도입해야 하는가?

로그를 원문 그대로만 남기면 검색과 집계가 어렵고 민감정보를 그대로 남길 위험도 커진다. 실무에서는 structured log로 `timestamp`, `level`, `service`, `route`, `requestId`, `traceId`, `release`, `userIdHash`, `errorCode`, `durationMs`, `message`, `stack`처럼 필드를 남긴다. 원문 stack trace는 보존할 수 있지만, 비밀번호, 토큰, 주민번호, 결제 정보 같은 민감정보는 절대 그대로 남기면 안 된다.

백엔드는 stdout/log collector/CloudWatch/ELK/Loki/Datadog으로 모으고, 프론트는 브라우저 console이 아니라 Sentry 같은 error monitoring, OpenTelemetry JS, RUM SDK, `window.onerror`, `unhandledrejection`, React Error Boundary로 운영 신호를 보낸다. 사용자 액션 수집은 별도다. GA4/GTM은 전환·퍼널·이벤트 분석, Hotjar류는 heatmap/session replay, Sentry는 에러와 성능, Web Vitals는 체감 성능을 본다. 하나만 고르는 문제가 아니라 목적별로 조합하고, PII 마스킹과 consent를 반드시 챙겨야 한다.

실무 조합은 아래처럼 나누는 것이 가장 안전하다.

| 목적             | 대표 도구                             | 프론트에서 남길 것                                                                             | 백엔드/운영에서 연결할 것                                                          | 주의점                                                             |
| ---------------- | ------------------------------------- | ---------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------------------ |
| 장애 원인 추적   | Sentry, OpenTelemetry, backend log    | `requestId`, `release`, `route`, `component`, `apiPath`, `httpStatus`, `errorCode`, breadcrumb | 같은 `requestId`를 가진 structured log, stack trace, DB/query 이름, upstream error | sourcemap upload 없으면 번들 stack trace가 읽기 어렵다.            |
| 사용자 행동 분석 | GA4, GTM                              | `sign_up_start`, `sign_up_success`, `search_submit`, `purchase` 같은 제품 이벤트               | 배포 버전, 실험군, campaign, route dimension                                       | GA4는 원인 추적 로그가 아니라 제품 분석 도구다.                    |
| 체감 성능 관측   | Web Vitals, RUM, Sentry Performance   | LCP, INP, CLS, route transition, API failure rate                                              | API p95/p99, CDN cache hit, DB latency, region                                     | 평균만 보지 말고 p95/p99와 기기·지역별 분포를 본다.                |
| UX 관찰          | Hotjar류 heatmap/session replay       | 클릭, 스크롤, 화면 이동, rage click, dead click                                                | 장애 시점의 requestId나 release와 느슨하게 연결                                    | 입력값 suppression, 개인정보 마스킹, sampling, 동의 정책이 먼저다. |
| 인프라 운영      | CloudWatch, Prometheus, Grafana, Loki | 보통 프론트가 직접 남기지는 않지만 route/release dimension은 연결 가능                         | CPU, memory, 5xx, queue depth, cache hit ratio, DB connection                      | 서버 메트릭만으로 사용자 체감 문제를 설명할 수 없다.               |

따라 할 수 있는 최소 구성은 이렇다.

1. 모든 API 응답에 `x-request-id`를 내려주고, 프론트 에러 UI와 Sentry 이벤트에 붙인다.
2. Sentry를 도입할 때 release, environment, sourcemap upload, React Error Boundary, unhandled promise rejection 수집을 같이 묶는다.
3. `web-vitals` 라이브러리로 LCP/INP/CLS를 route, release, device, network와 함께 GA4 또는 자체 endpoint로 보낸다.
4. GA4/GTM에는 제품 이벤트만 넣는다. 이벤트 이름과 parameter는 문서화하고, 임의 문자열을 여기저기 흩뿌리지 않는다.
5. Hotjar류 replay는 기본 수집을 켜는 것이 아니라 마스킹, suppression, sampling, 동의 정책을 먼저 정한 뒤 제한적으로 켠다.
6. 백엔드 로그는 requestId, traceId, status, durationMs, userIdHash, stack을 구조화해서 중앙 수집하고, 프론트의 requestId와 검색 가능하게 만든다.

#### 발췌 2. 메트릭을 잘 수집하면

네 질문은 `메트릭이 무엇인지`, `프론트와 백엔드 둘 다 하는지`, `호스트/종합 메트릭과 비즈니스 메트릭을 GA4와 어떻게 섞어 쓰는지`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. 메트릭은 로그처럼 사건을 자세히 남기는 것이 아니라 시간에 따라 변하는 숫자 지표인가?
2. 프론트 메트릭, 백엔드 메트릭, 인프라 메트릭, 비즈니스 메트릭은 각각 어떤 예시가 있고 어떤 도구로 수집하는가?
3. GA4는 메트릭을 다루지만 Prometheus/CloudWatch/Sentry RUM을 대체하지 않는 이유는 무엇인가?

메트릭은 시간에 따라 수집되는 숫자 지표다. `api_request_count`, `p95_latency_ms`, `cpu_usage`, `cache_hit_ratio`, `checkout_conversion_rate`, `DAU`처럼 이름, 값, 시간, label/dimension을 가진다.

프론트도 메트릭을 수집한다. Core Web Vitals, JS error rate, API failure rate, route transition time, checkout conversion rate가 예시다. 백엔드는 request rate, error rate, latency, queue depth, DB connection, cache hit ratio를 본다. GA4는 사용자의 행동 이벤트를 바탕으로 비즈니스/제품 메트릭을 보는 도구라서 `메트릭`을 다루는 것은 맞다. 하지만 서버 CPU, DB latency, queue depth 같은 운영 메트릭을 대체하지 않는다. 실무에서는 GA4, Sentry/RUM, Prometheus/CloudWatch를 목적별로 함께 쓰고 `requestId`, `release`, `route`, `region`, `experimentId` 같은 공통 차원으로 연결한다.

예를 들어 `회원가입 전환율이 떨어졌다`면 GA4만 보면 "어느 단계에서 이탈했는지"는 보이지만 "왜"는 약하다. 같은 시간대의 Sentry error rate, API 5xx, Web Vitals INP, backend p95 latency를 함께 봐야 한다. 반대로 Sentry만 보면 에러 원인은 보이지만 사업 영향도는 약하다. 그래서 운영에서는 `사용자 행동`, `에러 원인`, `체감 성능`, `서버 상태`를 분리 수집하고 공통 차원으로 다시 연결한다.

#### 발췌 3. 그림 1-19

네 질문은 `시스템 설계 그림을 어떤 태도로 봐야 하는지`, `면접에서 직접 설계 설명할 때 도움이 되는지`, `그림에 하나의 데이터센터만 포함했다는 말이 무엇인지`, `실제로는 어떻게 되는지`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. 시스템 설계 그림을 볼 때 박스 이름만 보는 것이 아니라 요청 흐름, 비동기 경로, 저장소, 관측 도구, 장애 경계를 어떤 순서로 읽어야 하는가?
2. 그림 1-19의 메시지 큐와 worker는 하나의 데이터 센터 안에 단순화되어 그려진 것인가, 실제 다중 데이터 센터에서는 지역별 큐/worker와 전역 관측 파이프라인이 어떻게 달라질 수 있는가?

그림은 반드시 선을 따라가야 한다. 먼저 경계선을 찾고, 사용자 요청의 동기 흐름을 따라가고, 큐 같은 비동기 흐름을 따로 표시하고, 저장소와 캐시를 찾고, 로그/모니터링/메트릭 같은 관측 도구를 찾고, 마지막으로 무엇이 생략됐는지 묻는다.

그림 1-19에서 지면상 하나의 데이터센터만 포함했다는 말은 DC1 안의 웹 서버, DB, 캐시, 큐, worker, 운영 도구만 단순화해 보여 준다는 뜻이다. 실제로는 각 데이터 센터마다 local queue와 worker가 있을 수 있고, 로그/메트릭은 지역별 수집 뒤 중앙 관측 시스템으로 모일 수도 있다. 전역 큐 하나를 둘지 지역별 큐를 둘지는 latency, 장애 격리, 데이터 주권, 운영 복잡도에 따라 달라진다.

#### 이 예시에서 쓰인 용어 풀이

- 로그: 에러, 요청 처리, 배포, 인증 실패처럼 나중에 원인을 찾기 위해 남기는 시간순 기록이다.
- structured log: JSON처럼 필드가 있는 형태로 남긴 로그다. 검색, 필터링, 집계가 쉽다.
- PII: 개인 식별 정보다. 로그와 리플레이에 그대로 남기면 보안 사고가 된다.
- error monitoring: 예외, stack trace, release version, 사용자 환경을 모아 원인 분석과 알림을 돕는 방식이다.
- 메트릭: 시간에 따라 수집되는 숫자 지표다.
- time series: 시간 순서로 저장되는 숫자 데이터다.
- label/dimension: route, region, browser, status_code처럼 메트릭을 쪼개 보는 기준이다.
- Core Web Vitals: LCP, INP, CLS 같은 핵심 웹 사용자 경험 지표다.
- GA4: 사용자 행동 이벤트, 전환, 유입, 재방문 같은 제품/비즈니스 분석에 강한 도구다.
- GTM: Google Tag Manager다. dataLayer에 넣은 이벤트를 GA4나 광고 태그 등으로 보내는 규칙을 관리한다.
- requestId: 하나의 요청을 프론트, API, DB, 작업 서버 로그에서 같이 찾기 위한 ID다.
- sourcemap: 번들링·압축된 JS 에러 위치를 원래 소스 파일과 라인으로 되돌려 보는 매핑 파일이다.
- session replay: 사용자의 화면 이동과 클릭 흐름을 재현하는 UX 관찰 도구다. 개인정보 마스킹이 필수다.
- breadcrumb: 에러 직전의 클릭, navigation, fetch 같은 작은 행동 단서다.
- observability: 로그, 메트릭, 트레이스 등을 통해 시스템 내부 상태를 외부에서 추론할 수 있는 능력이다.
- trace: 하나의 요청이 여러 서비스와 DB를 지나간 경로와 시간을 이어 붙여 보여 주는 관측 데이터다.

### 1장 · 38p 데이터베이스 규모 확장 도입

이번 배치는 데이터베이스 규모 확장 절의 시작이다. p38 하단 문장은 다음 페이지로 이어지므로, 여기서는 `DB도 무한히 커지지 않는다`와 `데이터 양 증가와 사용자 요청 증가는 다른 병목이다`까지만 잡는다.

#### 발췌 1. 저장할 데이터가 많아지면 데이터베이스에 대한 부하도 증가한다

네 질문은 `많은 데이터 때문에 내부에서 찾아내는 부하가 증가한다는 뜻인지`, `데이터가 많아진 만큼 사용자도 많아질 거라는 뜻인지`, `DB에도 저장 최대 용량이 있는지`, `수직 확장으로 무한히 만들 수 있는지`였다.

질문을 더 좋게 바꾸면 이렇게 된다.

1. DB 부하는 데이터 양 증가로 인한 저장/인덱스/백업/복제/working set 부담과, 사용자 증가로 인한 QPS/connection/write 부담을 어떻게 구분해야 하는가?
2. DB에는 엔진·스토리지·인스턴스·IOPS·connection 한계가 있으며, scale up과 storage autoscaling이 있어도 왜 무한 확장이 아닌가?
3. 데이터가 많아질 때 모든 조회가 단순히 선형으로 느려지는 것이 아니라 index, query plan, cache hit, disk I/O에 따라 달라지는 이유는 무엇인가?

네 생각은 방향이 맞다. DB는 무한대가 아니고 수직 확장으로도 무한히 키울 수 없다. 데이터 양이 늘면 테이블과 인덱스가 커지고, 자주 쓰는 working set이 메모리를 넘어서며, 백업·복제·vacuum/compaction·디스크 I/O 부담이 커진다. 사용자도 함께 늘면 읽기/쓰기 QPS, connection pool, lock contention이 증가한다.

인덱스가 있으면 모든 조회가 단순 선형 탐색은 아니지만, 인덱스도 저장 공간과 쓰기 비용을 먹고 메모리에 못 올라가면 latency가 커질 수 있다. DB에는 스토리지 최대치, CPU/RAM, IOPS, 네트워크, connection 수, 엔진별 테이블 크기 같은 한계가 있다. 그래서 어느 시점부터 read replica, partitioning, sharding 같은 수평 확장을 고민한다.

#### 이 예시에서 쓰인 용어 풀이

- working set: DB가 자주 읽는 데이터와 인덱스 일부다. 메모리에 들어가면 빠르고, 디스크로 밀리면 느려질 수 있다.
- index: DB가 원하는 행을 빨리 찾도록 돕는 별도 자료구조다. 찾기를 빠르게 하지만 저장 공간과 쓰기 비용을 추가로 쓴다.
- IOPS: 초당 디스크 입출력 횟수다. DB가 디스크에서 읽고 쓰는 작업량을 나타낸다.
- connection pool: DB 연결을 미리 만들어 재사용하는 묶음이다.
- replica lag: 주 DB에 반영된 변경이 읽기 복제본에 늦게 도착하는 시간 차이다.

## 2장 개략적인 규모 추정

- 프론트엔드 초점: 이미지, 동영상, 검색 API, 피드 렌더링이 결국 네트워크 비용과 저장소 비용을 만든다는 감각을 잡는다.
- 풀스택 확장: QPS, 저장소 용량, 대역폭, 가용성 수치를 먼저 추정해야 설계 토론이 추상적인 말장난으로 흐르지 않는다.

## 3장 시스템 설계 면접 공략법

- 프론트엔드 초점: 화면 컴포넌트 단위 설명이 아니라, 요구사항을 기능·성능·제약으로 분해하는 연습이 핵심이다.
- 풀스택 확장: 면접용 4단계 접근법은 실무 설계 문서에서도 그대로 재사용할 수 있다.

## 4장 처리율 제한 장치의 설계

- 프론트엔드 초점: 로그인, 검색, 좋아요, 댓글, 비밀번호 재설정, 업로드 같은 사용자 행동이 왜 제한되어야 하는지 본다.
- 풀스택 확장: fixed window, sliding window, token bucket 같은 알고리즘과 저장소 선택을 이해한다.
- 짧은 예시: 자동완성 입력창이 키 입력마다 요청을 날리면 프론트 최적화만으로는 부족하고, 서버 쪽 rate limit와 debounce 전략이 같이 맞물려야 한다.

## 5장 안정 해시 설계

- 프론트엔드 초점: 직접 해시 링을 구현할 일이 적더라도 캐시 서버나 샤드 분배가 왜 흔들리는지 이해해야 운영 이슈를 읽을 수 있다.
- 풀스택 확장: rehash 문제, 키 이동량, 균등 분포, 가상 노드 개념을 잡는다.

## 6장 키-값 저장소 설계

- 프론트엔드 초점: 세션, feature flag, 캐시, 좋아요 상태, 읽음 표시 같은 데이터가 왜 종종 키-값 저장소 위에 놓이는지 본다.
- 풀스택 확장: 단일 서버에서 분산 저장소로 갈 때 복제, 파티셔닝, 일관성, 장애 복구가 어떻게 등장하는지 본다.

## 7장 분산 시스템을 위한 유일 ID 생성기 설계

- 프론트엔드 초점: optimistic UI, 정렬, 임시 키, 목록 갱신, timestamp 기반 표시와 연결해서 본다.
- 풀스택 확장: Snowflake 계열 ID, 중앙 시퀀서, UUID, timestamp 기반 설계의 장단점을 비교한다.

## 8장 URL 단축기 설계

- 프론트엔드 초점: 공유 UX, 클릭 추적, 만료 링크, 리다이렉트 지연, 봇 트래픽을 함께 본다.
- 풀스택 확장: 해시 충돌, 중복 키, 읽기/쓰기 비율, analytics 확장 포인트를 본다.

## 9장 웹 크롤러 설계

- 프론트엔드 초점: 검색 인덱스와 추천 시스템이 결국 수집 파이프라인 위에서 만들어진다는 감각을 잡는다.
- 풀스택 확장: URL frontier, 중복 제거, politeness, robots, 저장 전략을 이해한다.

## 10장 알림 시스템 설계

- 프론트엔드 초점: 인앱 알림, 이메일, 푸시, 읽음 상태, 묶음 표시, 사용자 선호 설정이 UX 설계와 어떻게 연결되는지 본다.
- 풀스택 확장: fanout, channel abstraction, retry, delivery guarantee, template 시스템을 본다.
- 짧은 예시: 프론트에서 알림 배지만 잘 보여 주면 끝나는 게 아니라, 중복 발송과 순서 꼬임을 막는 백엔드 파이프라인이 함께 맞아야 한다.

## 11장 뉴스 피드 시스템 설계

- 프론트엔드 초점: 무한 스크롤, 초기 로딩, skeleton, pagination, 정렬 안정성, 읽음 상태가 피드 설계와 직결된다.
- 풀스택 확장: fanout-on-write vs fanout-on-read, 캐시, ranking, hot key 문제를 이해한다.

## 12장 채팅 시스템 설계

- 프론트엔드 초점: 실시간 메시지 반영, 타이핑 표시, 읽음 표시, reconnect, offline sync가 왜 복잡한지 본다.
- 풀스택 확장: WebSocket, polling, presence, ordering, message persistence, multi-device sync를 본다.
- 짧은 예시: "메시지가 늦게 보였다"는 이슈는 프론트 렌더링 지연이 아니라 소켓 연결 재수립, ack 처리, 서버 fanout 경로 문제일 수 있다.

## 13장 검색어 자동완성 시스템

- 프론트엔드 초점: debounce, cancel, stale response 방지, suggestion UI, 키보드 접근성과 연결해서 읽는다.
- 풀스택 확장: trie, prefix index, ranking, 인기 검색어 갱신, 다국어/오타 교정 전략을 본다.

## 14장 유튜브 설계

- 프론트엔드 초점: 썸네일, 플레이어 초기화, adaptive streaming, 업로드 진행률, 댓글/추천 영역의 결합을 본다.
- 풀스택 확장: transcoding, chunk upload, object storage, CDN, metadata pipeline, recommendation surface를 본다.

## 15장 구글 드라이브 설계

- 프론트엔드 초점: 업로드 상태 표시, 충돌 UX, 동기화 상태, 공유 링크, 폴더 탐색, 협업 경험을 본다.
- 풀스택 확장: chunk upload, metadata store, quota, revision history, sync engine, conflict resolution을 본다.
- 짧은 예시: 업로드 progress bar 하나 뒤에도 chunk 분할, 재시도, 백오프, 업로드 세션 관리가 숨어 있다.

## 16장 배움은 계속된다

- 프론트엔드 초점: 실제 서비스 아키텍처 글을 읽을 때 UI 증상과 시스템 구조를 연결하는 습관을 만든다.
- 풀스택 확장: 회사별 엔지니어링 블로그와 실제 장애/개선 사례를 통해 추상 개념을 현실 설계와 연결한다.
